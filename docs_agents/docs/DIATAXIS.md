# Diátaxis — 문서 분류 프레임워크 (docs-agents 정본)

> 모든 문서는 성격이 다르다. **착수 시 어느 분면인지 1줄로 정하고, 그 분면의 문체·구조를
> 따른다.** 이렇게 하면 "튜토리얼인데 레퍼런스처럼 건조", "설명인데 절차만 나열" 같은
> 성격 불일치를 막는다. (출처: Diátaxis 프레임워크 — gstack `/document-generate`가 채택)

## 1. 4분면 정의

문서는 **(학습 ↔ 작업)** 축과 **(실습 ↔ 이론)** 축으로 4가지 성격을 갖는다.

| 분면 | 지향 | 독자 상황 | 목적 | 문체 |
|---|---|---|---|---|
| ① 튜토리얼 (Tutorial) | 학습 지향 | "처음이라 따라 해보고 싶다" | 손을 잡고 끝까지 성공 경험을 준다 | 명령형·단계별·결과 예측 가능. 곁가지 설명 최소 |
| ② 하우투 가이드 (How-to) | 과업 지향 | "이 문제를 해결해야 한다" | 특정 목표를 달성하는 실용 레시피 | 목표→전제→단계. 이미 아는 독자 가정, 군더더기 없음 |
| ③ 레퍼런스 (Reference) | 정보 지향 | "정확한 사실을 찾는다" | 사실을 빠짐없이·정확히 기술 | 건조·일관·완결. 표·필드·ID 중심. 의견 없음 |
| ④ 설명 (Explanation) | 이해 지향 | "왜 이런지 배경을 알고 싶다" | 맥락·근거·트레이드오프를 설명 | 서술형·논증형. 대안과 이유, "왜"에 답함 |

- **혼동 주의:** 튜토리얼(학습용, 안전한 길)과 하우투(실무 문제 해결)는 다르다. 레퍼런스(사실)와
  설명(이유)도 다르다. 한 문서에 4성격을 섞지 말고, 필요하면 분리하거나 명확히 구획한다.

## 2. 우리 팀 산출물 ↔ Diátaxis 매핑

| 우리 문서 | 주 분면 | 비고 |
|---|---|---|
| 기획서 | ④ 설명 | 왜 만드는가·배경·목표 — 근거 서술 |
| 인사이트 분석 | ④ 설명 | 패턴·기회·적용 제안의 논증 |
| 레퍼런스 리서치 | ③ 레퍼런스(+④) | 사례·벤치마크 사실 정리. 해석은 인사이트로 분리 |
| 요구사항 정의서 | ③ 레퍼런스 | FR/NFR·ID·인수기준 — 사실·완결성 |
| 기능명세서 / 기술명세서 | ③ 레퍼런스 | 입력·처리·출력·규칙을 정확·완결하게 |
| API 명세서 / DB 설계서(ERD) | ③ 레퍼런스 | 엔드포인트·필드·스키마 — 건조·정확 |
| 화면정의서 / IA / 메뉴트리 | ③ 레퍼런스 | 구조·항목 사실 |
| 유저 플로우 / 시퀀스 / 와이어프레임 | ③ 레퍼런스(+④) | 흐름 사실. 의도 설명은 별도 |
| 사용 가이드 / 온보딩 | ① 튜토리얼 · ② 하우투 | 처음 사용=튜토리얼, 특정 작업=하우투 |
| 회의록 / 릴리즈노트 | (기록 Archive) | 4분면 밖의 시점 기록물. 사실·시간순 |

> 기록물(회의록·릴리즈노트)은 Diátaxis 4분면이 아니라 **시점 기록(Archive)** 성격이다.
> 분면을 억지로 끼우지 말고 사실·시간순·추적 가능성에 집중한다.

## 3. 운영 규칙

1. **문서 착수 시 분면을 1줄로 정한다.** 예: "이 문서는 ③ 레퍼런스 — 정확·완결·건조하게."
2. 그 분면의 **문체·구조**를 따른다(위 표). 레퍼런스에 의견을 섞거나, 튜토리얼에 곁가지
   설명을 늘어놓지 않는다.
3. 한 문서가 두 성격을 강하게 가지면 **분리**를 검토한다(예: API 레퍼런스 + "왜 이 설계인가" 설명 분리).
4. 검수(`/doc-review`·`/deep-review`) 시 "이 문서가 선언한 분면의 성격에 맞는가"를 점검 항목에 둔다.
