# 간트차트 + 칸반보드 (1개 스프레드시트 · 4개 시트)

> 담당 에이전트: `project-scheduler` · 슬래시 스킬: `/project-plan`
> 산출물: **하나의 Google 스프레드시트**에 아래 4개 시트(탭)를 만들어 서로 연동한다.
> 가독성·서식은 `docs/DOC-FORMATTING.md` **§0(색 토큰)·§3·§3-0(가독성 마무리)·§3-1(도구)**를 따른다.
> (이 템플릿은 간트/칸반 특유의 컬럼·연동·색 규칙만 정의하고, 셀 서식/가독성 마무리는 위 문서를 기준으로 한다. 중복 금지.)

## 0. 설계 원칙 (단방향 연동)

- **`Tasks`가 유일한 입력원(single source of truth).** `Gantt`·`Kanban`·`This Week`는 Tasks를
  **읽어서 표시만** 한다(단방향). 담당자·상태·날짜를 바꾸려면 **Tasks에서** 바꾼다.
- **한계(정직히 명시):** 시트 수식만으로는 **칸반 카드를 드래그해 Tasks 상태를 역으로 바꾸는
  양방향 실시간 반영은 어렵다.** 그래서 단방향(Tasks=입력원) 설계를 권장으로 못박는다.
- **추측 일정 금지.** 기간·담당·날짜를 모르면 셀에 `(확인 필요)`를 채우고 사용자에게 확인한다.
  `(확인 필요)`를 남길 땐 **누가(담당/검수자) 언제까지 확정할지**를 비고에 함께 적는다(미정의 책임자 방치 금지).
- **이 템플릿의 범위(가벼움 유지).** 문서팀의 일정 계획은 **보조 역량**이다. 컬럼·드롭다운 값·색 규칙·
  자가점검 수준으로 가볍게 돕는다. **자동 크리티컬패스 계산 엔진·리소스 레벨링·번다운 자동화·
  스토리포인트/속도(velocity)·무거운 RAID/리스크 레지스터는 넣지 않는다.** 크리티컬 패스는 **작성자 수동 강조**까지만.

## 1. `Tasks` 시트 — 마스터 데이터(입력원)

각 행 = 작업 1건. 헤더 1행 + 데이터 2행부터.

| 컬럼 | 설명 | 입력 방식 |
|---|---|---|
| `Task ID` | 작업 고유 ID (예: `T-001`) | 텍스트 |
| `대분류` | 에픽·마일스톤(예: `기획`, `백엔드`, `QA`, `스파이크/리서치`) | 드롭다운(데이터 유효성) |
| `작업명` | 작업 이름 | 텍스트 |
| `담당자` | 작업 담당 | **드롭다운(데이터 유효성)** — 자유입력 오타 방지 |
| `검수자` | 작업 검수 담당 | **드롭다운(데이터 유효성)** |
| `시작일` | YYYY-MM-DD | 날짜 |
| `종료일` | YYYY-MM-DD | 날짜 |
| `기간(일)` | 작업 기간 | 숫자(또는 종료-시작 수식) |
| `상태` | `대기` / `진행중` / `검수중` / `완료` / `지연` / `차단` | **드롭다운(데이터 유효성)** — 칸반·간트 색을 구동 |
| `진행률(%)` | 0~100 | 숫자 |
| `신뢰도` (선택) | 추정 신뢰도 `높음`/`중간`/`낮음` — fake precision 방지 | **드롭다운(데이터 유효성)** |
| `의존(선행 Task ID)` | 선행 작업 ID(쉼표 구분) | 텍스트 |
| `완료조건(DoD)` (선택) | 완료로 인정할 조건. `대분류=개발`은 "코드리뷰 통과·빌드/린트 통과" 권장 | 텍스트 |
| `비고` | 메모(차단사유·차단원인 Task ID·`(확인 필요)` 책임자 등) | 텍스트(긴 열 → 줄바꿈+행높이) |

- **드롭다운 값(권장):**
  - 상태: `대기`, `진행중`, `검수중`, `완료`, `지연`, `차단`
  - 신뢰도: `높음`, `중간`, `낮음`
  - 담당자/검수자: 팀원 명단 `(TBD - 작성 필요)` — 사용자에게 받아 데이터 유효성 목록으로.
- **차단(Blocked) 규칙:** 상태=`차단`이면 **비고에 `차단사유`와 `차단원인 Task ID`를 필수로 적는다**
  (무엇이 풀려야 진행되는지 명시). `지연`(종료일 경과)과 `차단`(선행/외부요인 막힘)은 다른 상태다.
- **검수중 정의:** `검수중`은 **코드리뷰 + QA를 포함**하는 단계다. QA 단계는 **QA_agents 팀으로 핸드오프**
  하는 작업이면 비고/대분류(`QA`)에 그렇게 표기한다. (상태를 `리뷰중/QA중`까지 더 쪼개지 않는다 — 가벼움 유지.)
- **신뢰도=낮음** task는 추정 전 **스파이크/리서치 1건을 선행**으로 두길 권장한다(대분류 `스파이크/리서치`).
- **(확인 필요) 책임자 명시:** 기간·담당·날짜를 `(확인 필요)`로 남기면 비고에 **누가 언제까지 확정**할지 적는다.
- 의존 역전(선행이 더 늦게 끝남)·날짜 충돌이 있으면 작성 시 사용자에게 알린다.

## 2. `Gantt` 시트 — 타임라인 시각화

- **행** = 작업(`Tasks` 참조: Task ID·작업명·담당자·검수자가 보이게). **열** = 날짜.
- **기본 주 단위 열**(프로젝트 전체를 한눈에). 옵션으로 일 단위 열도 가능.
- **막대(bar):** 작업의 시작~종료 구간 셀을 **상태별 색**으로 칠한다. 가능하면
  **조건부서식/수식**으로 `Tasks`의 시작일·종료일·상태를 읽어 **자동 색칠**(수동 색칠 최소화).
- **이번 주 열 강조(필수):** 이번 주에 해당하는 열은 배경색/세로 라인으로 강조 →
  "지금 무슨 일이 진행 중"이 즉시 보이게. (`docs/DOC-FORMATTING.md §3-0`의 이번주 강조 규칙 적용.)
- **크리티컬 패스 수동 강조:** 여유(slack)가 0으로 이어지는 연쇄(가장 빡빡한 경로)는 **작성자가 막대를
  굵게/테두리 강조**해 표시한다. **자동 계산은 하지 않는다**(수동 강조까지만).

### 상태별 막대 색 규칙

| 상태 | 색(HEX) | 비고 |
|---|---|---|
| 대기 | `#D0D5DD` (구분선 회색) | 아직 시작 안 함 |
| 진행중 | `#1F3A5F` (포인트 네이비) | 현재 작업 |
| 검수중 | `#EEF2F7` 배경 + 네이비 테두리 | 검수 단계(코드리뷰+QA 포함) |
| 완료 | 연한 녹색 `(TBD - 작성 필요)` 또는 체크 표시 | 종료 |
| 지연 | `#C0392B` (위험 빨강) | 종료일 경과·미완료 |
| 차단 | `#E67E22` (주황) 또는 회색 빗금 | 선행/외부요인에 막힘. **빨강(지연)과 시각 구분.** 비고에 차단사유·차단원인 Task ID |

> 정확한 조건부서식/수식 도구명·시그니처에 확신이 없으면 `(확인 필요: 정확한 도구명/파라미터)`로
> 표기한다. 레이아웃·주 단위 열·이번주 강조·색 규칙은 도구와 무관하므로 단정해서 지킨다.

### 일정에 리뷰·재작업 버퍼 반영 (개발 task 권고)

- **개발 task는 `검수중` 단계가 기간을 갖도록** 잡는다(코드리뷰·QA에서 막판 Blocker가 나오면 재작업 시간이 든다).
- 큰 구현 task는 **리뷰·재작업 버퍼(구현 기간의 약 15~20%)** 를 기간에 포함하거나 **별도 행**으로 둔다.
- 예: 구현 10일 → 검수중 2일(리뷰·재작업) 별도 행, 또는 기간을 12일로 잡고 비고에 "리뷰·재작업 버퍼 포함" 표기.

### 의존 지연 전파(경량·수동)

- **자동 전파 계산은 하지 않는다.** 대신 작성/갱신 시: **선행 task의 종료일이 바뀌면 그 후행 task의
  시작일에 미치는 영향을 1줄로 경고**한다(예: "T4 종료일이 2일 밀려 후행 T5 시작일 검토 필요").

## 3. `Kanban` 시트 — 상태별 보드

- **컬럼 = 상태:** `대기` / `진행중 (WIP≤2/인)` / `검수중` / `차단` / `완료`.
  - **`진행중` 헤더에 WIP 한도를 표기**한다(예: `진행중 (WIP≤2/인)`). 같은 담당자가 동일 기간에
    `진행중` 2건 이상이면 작성/갱신 시 **사용자에게 알린다**(과부하·동시 진행 과다 방지).
  - **`차단` 컬럼**: 상태=`차단`인 카드를 모은다. 카드에 차단사유·차단원인 Task ID가 보이게.
- **`Tasks`에서 상태별로 자동 분류:** 각 컬럼은 `FILTER`/`QUERY` 수식으로 `Tasks`에서
  `상태=해당값`인 행을 끌어온다. → **Tasks에서 담당자·상태만 바꾸면 칸반이 자동 갱신.**
- **카드 표시 항목:** 작업명 · 담당자 · 검수자 · 기간(또는 종료일). "이번 주 누가 무슨 Task 중인지"
  + 전체 진행 상황을 바로 확인.
- **단방향:** 칸반은 **읽어서 표시**만 한다. 상태 변경은 `Tasks`에서.

### 칸반 컬럼 수식 예 (확인 필요: 정확한 함수 표기/도구)

```
=FILTER(Tasks!C2:E, Tasks!I2:I = "진행중")   # 작업명·담당자·검수자 등 표시 (열 범위는 실제 시트에 맞춤)
```
> `FILTER`/`QUERY`의 정확한 인자·시트 범위는 실제 `Tasks` 컬럼 위치에 맞춰 조정한다.
> 수식 입력 도구가 불확실하면 `(확인 필요)` 표기 + §3-1-E 폴백(값은 채우고 수식은 사용자에게 안내).

## 4. `This Week` 시트 — 주간 포커스 (선택)

- 이번 주에 **진행/검수 예정**인 작업만 `FILTER`로 모아 보는 뷰(시작일~종료일이 이번 주와 겹침).
- 전체 조망은 `Gantt`/`Kanban`, **이번 주 포커스**는 이 시트로.
- 컬럼(권장): `Task ID` / `작업명` / `담당자` / `검수자` / `상태` / `종료일` / `진행률(%)`.

```
=FILTER(Tasks!A2:L, (Tasks!F2:F <= 이번주_종료) * (Tasks!G2:G >= 이번주_시작))   # (확인 필요: 정확한 함수/범위)
```

## 5. 가독성·서식 (필수 — `docs/DOC-FORMATTING.md` 준수)

작성 마지막에 **반드시** 아래를 적용한다(상세는 `docs/DOC-FORMATTING.md §3-0/§3-1`):

- 헤더 1행: 네이비 배경(`#1F3A5F`) + 흰 굵은 글씨 + 가운데 정렬 + **틀 고정**.
- **열 너비 자동맞춤**(헤더보다 좁지 않게), 긴 열(비고 등)은 **줄바꿈(wrap) + 행 높이 자동확장 세트**.
- 세로 정렬 **상단(top)**, 가로: 텍스트=왼쪽 / 숫자·상태·진행률=가운데.
- 데이터 영역 **셀 병합 금지**(필터·정렬·줄바꿈 깨짐 방지).
- **셀이 글보다 작아 잘리지 않는지** 확인하고 끝낸다.
- `문서정보`(또는 `Tasks` 상단 메타): 프로젝트명/버전/작성자/작성일/상태 `(TBD - 작성 필요)`.

## 6. 채울 항목 체크 (작성 시)
- [ ] 팀원 명단(담당자·검수자 드롭다운 값) `(TBD - 작성 필요)`
- [ ] 작업 목록(WBS) — 상위 문서(기획/요구사항/기능명세)에서 도출 or 사용자 입력
- [ ] 각 작업의 기간·시작/종료일·의존 — 모르면 `(확인 필요)`
- [ ] 프로젝트 기간(간트 날짜 범위) `(TBD - 작성 필요)`
- [ ] **개발 task에 리뷰·재작업 시간**(검수중 기간 또는 구현의 ~15~20% 버퍼)이 일정에 있는가
- [ ] **차단 상태가 가시화**됐는가(차단 컬럼/색) — 차단 task에 차단사유·차단원인 Task ID가 있는가
- [ ] **1인 동시 `진행중`이 WIP 한도(≤2)를 넘지 않는가** — 넘으면 사용자에게 알렸는가
- [ ] 불확실 task에 **신뢰도(높음/중간/낮음)** 와 `(확인 필요)`+**검증 책임자·기한**이 적혔는가
- [ ] 크리티컬 패스(여유 0 연쇄)를 작성자가 **수동 강조**했는가 / 종료일 변경 시 후행 영향을 1줄 경고했는가
