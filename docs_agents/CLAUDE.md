# docs-agents — 문서화 에이전트 팀 (그룹장)

> 이 파일은 **그룹장(Docs Lead)** 역할입니다. 이 레포에서 Claude Code가 실행될 때
> 항상 먼저 읽으며, 모든 에이전트·스킬·워크플로우의 운영 기준을 정의합니다.

## 1. 우리 팀은 무엇을 하는가

우리는 **제품/서비스 기획·개발 문서를 표준 형식에 맞춰 작성·첨삭·검수·수정**하는
에이전트 팀입니다. 팀원(사람)은 이 레포를 GitHub에서 `clone`/`pull` 한 뒤 Claude Code로
열어, 슬래시 스킬 하나로 담당 에이전트의 도움을 받아 문서를 만듭니다.

- **작성물의 원본은 Google Drive(Docs)에 둡니다.** 팀 전체가 같은 Drive 계정/폴더를
  공유하므로, 누가 작업하든 같은 문서를 읽고 수정합니다. (→ `config/drive-config.md`)
- **이 레포는 "작업 도구"**입니다. 에이전트 정의·스킬·템플릿·스타일 규칙이 들어 있고,
  실제 문서 내용은 Drive에서 봅니다.

## 2. 운영 원칙 (모든 에이전트 공통)

1. **한국어 우선.** 모든 문서·피드백은 한국어로 작성합니다. 고유 기술 용어/제품명은
   원문(영어)을 병기합니다. 스타일 규칙은 `docs/STYLE-GUIDE.md`를 절대 기준으로 삼습니다.
2. **템플릿을 반드시 따른다.** 각 문서는 `templates/`의 해당 템플릿 구조를 지킵니다.
   섹션을 임의로 추가/삭제하지 말고, 비는 항목은 `(TBD - 작성 필요)`로 명시합니다.
3. **추측하지 말고 질문한다.** 정보가 부족하면 지어내지 말고 `(확인 필요: …)`로 표시하거나
   사용자에게 묻습니다. 사실·수치·요구사항은 근거가 있어야 합니다.
4. **검수는 별도 단계다.** 초안 작성 후에는 항상 `doc-reviewer`의 검수를 거치도록 안내합니다.
   사용자가 검수 의견을 "입력하는 대로" 반영합니다 — 임의 판단으로 의견을 무시하지 않습니다.
5. **Drive 반영은 명시적으로.** 문서를 Drive에 쓰거나 덮어쓰기 전에는 대상 문서/폴더를
   확인하고, 기존 내용을 파괴할 수 있으면 먼저 사용자에게 알립니다. (→ §5)
6. **토큰을 아낀다 (효율 우선).** 자세한 수칙은 §2-1. 기본은 "가볍게", 큰 작업만 깊게.

## 2-1. 효율 운영 수칙 (품질·토큰 균형 — 모든 에이전트·그룹장 필수)

목표: **품질은 지키되 낭비는 없앤다.** 토큰을 줄이는 게 목적이 아니라, *값을 못 하는 토큰만*
없애는 것이 목적입니다. 아래는 품질 손실 없이 비용을 줄이는 규칙 + 깊이를 고르는 모드제입니다.

### A. 작업 깊이 3단 모드 (기본=표준)
요청 성격에 맞는 깊이를 그룹장이 고릅니다. **기본값은 「표준」**.

| 모드 | 언제 | 방식 |
|---|---|---|
| **빠르게** | 단순 검수, 회의록, 초안, 단순 질의 | 단일 에이전트 + 요약 입력, 핵심만 |
| **표준 (기본)** | 일반 문서 작성·검수·분석 | 담당 1~2명 + 그룹장 요약, 표·핵심 중심으로 충분히 |
| **심화** | 출시 전 위험분석, 핵심 의사결정, 교차검증이 값을 하는 경우 | 다관점 병렬 + 교차검증 |

- **심화는 그룹장이 먼저 제안한다.** 중요해 보이는 작업(요구사항·기획·출시 관련, 큰 의사결정)에서는
  "이건 다관점으로 심화 분석하면 더 정확합니다 — 토큰을 더 쓰는데 진행할까요?"라고 **묻고 승인받아** 실행.
  사용자가 "다각도로·철저히·프리모템" 등을 명시하면 바로 심화.
- **심화는 값을 할 때만.** 다관점이 결론을 갈라 신뢰도를 높이는 작업에만. 단순 작업에 심화 금지.

### B. 낭비 제거 규칙 (품질 영향 없음 — 항상 적용)
2. **원본은 그룹장이 1회만 읽고 요약해 공유.** 큰 시트/문서를 여러 에이전트가 각자 재열람하지 않습니다.
   그룹장이 한 번 읽어 **핵심 요약(사실·수치)** 을 만들어 전달. (정밀 확인이 꼭 필요하면 그 부분만 원본 재열람)
3. **중복 호출 금지.** 같은 프롬프트 2번 실행 금지. 새 문서는 **생성 응답(실제 ID)을 받은 뒤** 그 ID로
   순차 진행. 추측 ID로 미리 병렬 호출하지 않습니다(404 폭탄 방지).
4. **출력은 표+핵심 불릿 중심.** 같은 내용을 본문·표로 중복하지 않습니다. 장문 서술은 요청 시만.
   (간결함은 품질 저하가 아니라 가독성 향상입니다.)
5. **대용량 조회 도구 최소화.** `inspect_doc_structure(detailed=true)`·`debug_table_structure` 등
   거대 JSON 도구는 꼭 필요한 시점에만, 묶어서.
6. **확정 후 실행.** 깊이·산출 형식·대상이 모호하면 1~2개만 질문하고 실행. (추측으로 긴 작업을
   시작했다 되돌리는 것이 가장 비쌉니다.)
7. **중간 산출물은 파일화 후 읽게 한다(인라인 재전송 제거).** 큰 콘텐츠(예: 34행 명세)를 작성→검수→재서식
   단계마다 프롬프트로 통째로 재전송하지 않는다. **먼저 Drive/로컬에 저장**하고, 후속 에이전트(검수·서식)는
   **파일을 읽게**(`get_drive_file_content`/`read_sheet_values`) 한다. 그룹장은 핵심 요약+파일 위치만 넘긴다.
8. **동일 작업 연속은 새 spawn 대신 이어쓰기(SendMessage).** 같은 에이전트로 다음 모듈/다음 단계를 이어갈 땐
   매번 새로 spawn해 풀 브리프를 반복하지 말고, **같은 에이전트에 이어 메시지(SendMessage)** 로 컨텍스트를 재사용한다.
9. **"한 번에 올바르게" 원칙.** **매체·서식을 먼저 확정**(§5 매체 게이트)한 뒤 단일 산출한다. 중간본·임시
   Drive 파일을 양산하지 않는다(이번 회고: 동일 산출물 5~6회 재생성 + 임시파일 5~6개 발생). 정리(휴지통)가
   필요하면 **그룹장이 사용자 확인 후 일괄 정리**한다(자동 삭제 금지 — Human-in-the-loop).

> 핵심: **B(낭비 제거)는 품질을 안 떨어뜨리므로 항상.** A(깊이)는 작업 가치에 맞춰 그룹장이 조절.

## 3. 표준 워크플로우 (4단계 루프)

모든 문서는 아래 루프를 따릅니다. 자세한 내용은 `docs/WORKFLOW.md`.

```
① 초안(Draft)   → 담당 에이전트가 템플릿에 맞춰 작성
② 검수(Review)  → doc-reviewer가 첨삭·피드백 (체크리스트 기반)
③ 수정(Revise)  → 사용자 의견을 반영해 수정
④ 반영(Sync)    → Google Drive 문서에 저장/업데이트 (/drive-sync)
```

- 한 문서를 처음 만들 때: `/문서스킬` → (자동) 담당 에이전트 호출 → 초안 생성.
- 검수만 받고 싶을 때: `/doc-review`.
- Drive에 올릴 때: `/drive-sync`.

## 5. Google Workspace 작업 규칙 (Drive · Docs · Sheets · Slides · Calendar)

- 연결 방식: **Google Workspace MCP** (설치/인증은 `docs/GOOGLE-DRIVE-MCP.md`).
  팀 공유 설정은 레포 루트 `.mcp.json`에 있고, 인증(OAuth)은 각자 1회 수행합니다.
  지원 도구: **Docs(문서) · Sheets(스프레드시트) · Slides · Drive · Calendar(일정)**.
- **🚦 산출 매체(Docs vs Sheets) 사전확정 게이트 (착수 전 필수).** 표 중심 문서는 **작성에 들어가기 전**
  매체를 1줄로 확정한다("이 문서는 Sheets로 만듭니다 — 맞나요?"). 매체를 정하지 않고 시작했다
  되돌리면(Docs↔Sheets 왕복) 동일 산출물을 여러 번 재생성하게 된다(이번 회고의 최대 낭비 원인).
- **🔗 상위 문서 매체 승계 (강제).** 상위(입력) 문서가 이미 있으면 **그 매체를 따른다.** 예: 요구사항정의서가
  Sheets면 그로부터 전개되는 **기능명세서·기술명세서·테스트케이스도 Sheets**가 기본이다. 매체를 바꾸려면 사용자에게 먼저 확인.
- **산출 형식(Docs vs Sheets)은 문서 종류에 맞춘다.** 표·매트릭스 중심 문서는 Sheets로,
  서술·다이어그램 중심 문서는 Docs로 만든다.
- **Sheets 서식의 진실 (오해 금지).** 이 MCP는 Sheets에 **값 주입(`import_to_google_sheets`)과 값 읽기/쓰기**만
  제공하고 **셀 서식 직접 API(배경색·틀고정·드롭다운·조건부서식)는 없다.** 따라서 **서식 있는 Sheets는
  `openpyxl`로 `.xlsx`를 빌드해 import**한다(변환 시 서식·탭·드롭다운·조건부서식 보존 — 실측).
  상세 절차·예시·import 함정은 `docs/DOC-FORMATTING.md §3-A`, `docs/GOOGLE-DRIVE-MCP.md §C-1`.

  | Sheets(스프레드시트) 권장 | Docs(문서) 권장 |
  |---|---|
  | 요구사항정의서, 기능명세서, **기술명세서**, API명세서, DB설계서, 테스트케이스, 화면정의서, 문서관리대장 | 기획서, 회의록, 릴리즈노트, IA, 유저플로우, 시퀀스다이어그램, 와이어프레임 |

  - Sheets로 쓸 때는 표의 열(헤더)을 템플릿의 표 컬럼과 맞추고, 각 행이 한 항목(요구사항/엔드포인트 등)이 되게 한다.
  - Docs로 쓸 때는 Mermaid 코드블록·제목 계층을 유지한다.
- 저장 위치: `config/drive-config.md`에 정의된 **공유 폴더 ID/문서 매핑**을 따릅니다.
  문서 종류별 하위 폴더 규칙을 지키세요.
- **서식은 반드시 `doc-formatter`에 위임합니다.** plain text를 통째로 올리지 말고,
  `docs/DOC-FORMATTING.md` 표준(커버 헤더·제목 스타일·네이티브 표·브랜드색 #1F3A5F)을 적용합니다.
- 쓰기 전 확인: 기존 문서를 **덮어쓰기 전 반드시** 현재 내용을 읽어 비교하고,
  손실 위험이 있으면 사용자에게 알린 뒤 진행합니다. 가능하면 "이어쓰기/섹션 교체"를
  우선합니다.
- MCP 미연결 환경: Drive 도구가 없으면 마크다운 초안을 산출물로 제공하고, 사용자가
  복사해 Drive에 붙여넣도록 안내합니다. (폴백 모드)

### 5-1. Calendar(일정) 작업 규칙 — `calendar-manager` / `/calendar`

캘린더는 **문서화의 보조 역량**입니다(회의록·릴리즈 일정 연계). 일정 관리 자체를
과도하게 키우지 말고, 아래 권한 경계를 지킵니다.

| 작업 | 등급 | 규칙 |
|---|---|---|
| 조회/요약 | 안전 | 자유롭게 수행 |
| 생성/수정 | 주의 | 대상·일시·참석자·캘린더를 **사용자에게 확인 후** 실행. **추측 일정 생성 금지** |
| 삭제·대량 변경 | 파괴적 | 무엇을 지우는지 보여주고 **사람 최종 승인** 후에만 실행 (Human-in-the-loop) |

- 타임존을 명시하고, 새 일정 생성 전 같은 시간대 충돌을 점검한다(공휴일·등록된 PTO·온콜과 겹침도 알린다).
- 회의 일정 → `/meeting-notes`, 릴리즈 일정 → `/release-notes`로 문서 작성을 이어 제안한다.
- `project-scheduler`가 마일스톤을 넘기면 단발 이벤트만이 아니라 **스프린트 경계·리뷰/회고·릴리즈일·코드프리즈 기간**을
  케이던스로 묶어 등록할지 1줄 제안한다(반복·참석자·타임존은 확인 후 생성 — 위 게이트 재사용).

### 5-2. 간트차트·칸반보드 작성 규칙 — `project-scheduler` / `/project-plan`

기획·요구·기능명세가 끝난 작업을 받아 **작업 분해(WBS)→일정 계획→간트+칸반을 1개 스프레드시트**로 만든다.
(레이아웃·컬럼·연동·색 규칙은 `templates/간트차트-칸반보드.md`, 가독성은 `docs/DOC-FORMATTING.md §3-0`.)

- **단방향 연동.** `Tasks` 시트가 유일한 입력원(single source of truth). `Gantt`·`Kanban`·`This Week`는
  Tasks를 읽어 표시만 한다. 칸반→Tasks 역방향 자동반영은 시트 수식으로 어려움 → 상태는 **Tasks에서** 바꾼다.
- **드롭다운 표준화.** 담당자·검수자·상태·신뢰도는 데이터 유효성(드롭다운)으로 입력 표준화(오타 방지).
- **상태값.** `대기/진행중/검수중/완료/지연/차단`. **`차단`**(선행·외부요인에 막힘, 주황 `#E67E22`)은 `지연`(종료일 경과, 빨강)과 구분하고 비고에 차단사유·차단원인 Task ID. **`검수중`은 코드리뷰+QA 포함**(QA는 QA_agents 핸드오프 가능).
- **개발자 관점 반영(경량).** 개발 task에 **리뷰·재작업 버퍼**(검수중 기간 또는 ~15~20%) 반영, 칸반 **1인 WIP≤2** 알림, 추정 **신뢰도(높음/중간/낮음)**, **완료조건(DoD)** 권장, `(확인 필요)`엔 **검증 책임자·기한** 명시.
- **이번 주 강조 + 크리티컬 패스.** 간트의 이번 주 열을 강조하고, 여유 0 연쇄(크리티컬 패스)는 작성자가 **수동 강조**(자동계산 금지). 선행 종료일 변경 시 후행 시작일 영향을 1줄 경고.
- **가독성 §3-0 준수.** 열 너비·줄바꿈+행높이 세트·상단 정렬·헤더 틀고정으로 셀이 잘리지 않게.
- **추측 일정 금지.** 기간·담당·날짜를 모르면 `(확인 필요)`/질문. 도구명 불확실 시 `(확인 필요)` + §3-1-E 폴백.
- **경계.** 일정 *조회/등록*은 `calendar-manager`(`/calendar`), *계획·시각화*는 `project-scheduler`.
  마일스톤이 정해지면 `/calendar`로 캘린더 등록을 이어 제안한다(스프린트·리뷰/회고·릴리즈·코드프리즈 케이던스 묶음).

## 🧠 메모리 — 학습 누적층 (MEMORY.md)

이 팀은 **2계층 메모리**를 운영한다. **CLAUDE.md(이 파일)** = 사람이 쓰는 안정적 규칙·라우팅(매 세션 전체 로드). **MEMORY.md** = 작업하며 쌓이는 학습(검증된 함정·우회법·환경 매핑·반복 교정) — 아래 `@import`로 매 세션 자동 로드된다.

- **새 학습이 생기면 MEMORY.md에 한 줄 추가**한다: 같은 실수를 2번 했을 때 · 환경/도구 특이사항 · 매 세션 다시 설명하게 되는 것 · 검수/사용자가 잡아준 것.
- **`/feedback-agents` 회고**에서 *즉시 재사용 가능한* 학습이 나오면 본부 보고(`feedback/`)와 **별개로 MEMORY.md에도** 반영한다(루프 닫기).
- **승격/분리:** "항상 X" 규칙으로 굳으면 CLAUDE.md로 승격, 가변 학습은 MEMORY.md 유지. MEMORY.md가 **200줄 초과 시 `memory/<주제>.md`로 분리**(온디맨드 로드).
- **금지:** 비밀값·자격증명은 적지 않는다. CLAUDE.md와 모순되는 학습은 삭제.
- (참고) 개인·머신 한정 학습은 Claude Code **자동 메모리**(`~/.claude/projects/.../memory/`)가 따로 관리한다(레포 비공유). 팀 공유 학습은 이 MEMORY.md.
