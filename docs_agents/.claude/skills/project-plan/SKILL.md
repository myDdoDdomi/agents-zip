# /project-plan — 일정 계획 · 간트차트 + 칸반보드

`project-scheduler` 에이전트를 사용해 **작업 분해(WBS) → 일정 계획 → 간트 + 칸반 스프레드시트**를 만듭니다.

대상: **$ARGUMENTS** (프로젝트명, 또는 입력이 될 기능/요구사항/기획 문서)

## 절차
1. **입력 확보.** `$ARGUMENTS`로 받은 프로젝트/기능의 상위 문서(기획서·요구사항·기능명세)가
   Drive에 있으면 참조해 작업 목록을 도출한다. 없으면 작업·기간·담당을 사용자에게 받는다.
   **기간·담당·날짜를 모르면 추측하지 말고 `(확인 필요)`로 표기하거나 1~2개만 질문한다.**
2. **`project-scheduler`에게 위임.** `templates/간트차트-칸반보드.md` 구조로 1개 스프레드시트에
   4개 시트(`Tasks`/`Gantt`/`Kanban`/`This Week`)를 만든다.
   - **`Tasks` = 입력원(single source of truth)**, 나머지는 읽어서 표시(단방향).
   - 담당자·검수자·상태는 **드롭다운(데이터 유효성)**. 상태가 칸반·간트 색을 구동.
   - **이번 주 열 강조** + 상태별 막대 색(지연=빨강).
3. **가독성 마무리(필수).** `docs/DOC-FORMATTING.md §3-0` — 열 너비·줄바꿈+행높이 세트·상단 정렬·
   헤더 틀고정으로 **셀이 잘리지 않게** 한다.
4. **연계.** 마일스톤·핵심 일정이 정해지면 `/calendar`(`calendar-manager`)로 캘린더 등록을 이어 제안.

> **한계(정직히):** 시트 수식만으로 칸반→Tasks 역방향 자동반영(드래그로 상태 변경)은 어렵습니다.
> 상태는 **Tasks에서 바꾸면** 칸반·간트가 따라 갱신됩니다(단방향).
> 수식/조건부서식/드롭다운 도구명이 불확실하면 `(확인 필요)`로 표기하고, 값은 채운 뒤
> 미적용 자동화를 사용자에게 수식으로 안내합니다(§3-1-E 폴백). 거짓 도구명을 지어내지 않습니다.
