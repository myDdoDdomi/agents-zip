# /deep-review — 다관점 심화 검수

중요 문서를 **5개 렌즈로 병렬 검수**한 뒤 `doc-reviewer`가 **1개 판정으로 종합**합니다.
CLAUDE.md §2-1의 「심화」 모드를 실제 동작으로 구현한 것입니다. 새 에이전트 없이 기존
에이전트를 렌즈로 활용합니다.

## 게이트 (먼저 승인)
심화는 비용이 큽니다. **그룹장이 먼저 제안·승인** 후 실행합니다(§2-1 A). 일반 검수는 `/doc-review`.
출시 전 위험분석·핵심 의사결정·교차검증이 값을 하는 문서에만 씁니다.

## 절차
1. 그룹장이 검수 대상을 1회 읽어 핵심 요약을 만들고, 5개 렌즈에 **병렬**로 전달한다(각자 재열람 금지, 정밀 확인 필요분만 원본).
2. **5개 렌즈** — 각 렌즈는 발견을 **신뢰도·심각도와 함께 전량 보고**(필터는 종합 단계 — Opus 4.8 기준):

   | 렌즈 | 관점 | 점검 초점 |
   |---|---|---|
   | ① 기획 타당성·전략 | doc-planner 관점 | 문제·목표·범위가 타당한가, 성공지표 측정 가능한가 |
   | ② 요구 정합·추적성 | requirements-analyst 관점 | 요구↔기능↔화면/API/TC ID 누락·끊김 |
   | ③ 기술 실현성·엣지케이스 | spec-writer/api-designer 관점 | 구현 가능성, 예외·경계·에러 케이스 누락 |
   | ④ 사용자 가치·UX | ux-architect 관점 | 실제 사용자 가치·흐름·사용성 |
   | ⑤ 일관성·용어·측정가능성·스타일 | doc-reviewer 체크리스트 | 용어·ID 통일, 모호어, 스타일 가이드, 선언 분면 일치(docs/DIATAXIS.md) |

3. **종합** — `doc-reviewer`가 5개 렌즈 보고를 받아 중복 제거 → 우선순위 → **1개 판정**(통과 / 조건부 / 보류)으로 합친다. 각 지적은 `위치|심각도|문제|근거|수정 제안` 표로.
4. 사용자가 승인하면 지시한 대로만 반영하고 변경 요약을 보고한다. 이어서 `/drive-sync` 안내.

## 주의
- 렌즈 단계에서 "고심각만 보고" 같은 필터를 걸지 않는다 — 전량 보고 후 종합 단계에서 우선순위화한다.
- 임의로 의미를 바꾸지 않는다. 제안은 제시하되 채택은 사용자가 결정한다.
