## 2. 이슈 상세 (현상 -> 영향 -> 권장 업데이트)

### A. 높음 — 코어 파이프라인 silent catch가 치명적 실패를 영구 은폐

- **현상:** `profile.py`의 `trigger_recompute_after_analysis`는 `_do_recompute` 호출 실패 시 `except Exception`으로 잡아 `logger.warning`만 남기고 분석 잡 상태에는 영향을 주지 않도록 설계돼 있다(코드 확인: `profile.py` L314–324). 이 설계 자체는 "분석 잡 실패 방지"라는 의도가 있었으나, 결과적으로 `/profile/compute` 422(필수 필드 `color` 누락) 에러가 **매 분석 완료마다 삼켜지면서 성향이 한 번도 생성되지 않았다.** 운영 로그를 직접 추적하기 전까지 누구도 알 수 없었다. 수정 후 현재 코드(`profile.py` L315–324)에는 `exc_info=True` 로깅이 추가돼 스택트레이스가 남는다.
- **영향:** best-effort 패턴이 "보조 경로(부가 기능)"가 아닌 "코어 데이터 생성 경로"에 적용되면, 실패가 침묵 속에 누적된다. 홈 대시보드가 늘 비어 있던 근본 원인이었고, 단위·정적 검사로는 탐지 불가능했다. dev-lead 계획·code-reviewer 게이트 두 단계를 통과했다.
- **권장 업데이트:**
  - `agents/code-reviewer.md` — 리뷰 게이트에 항목 추가: "코어 데이터 생성 경로(성향·분석·집계 등 제품 핵심 기능의 출력을 만드는 경로)에 bare `except Exception`이 있으면, 실패가 침묵 속에 누적될 수 없는지 확인한다. best-effort가 허용되는 경우는 '보조 경로(라벨 추가·알림 등)' 에 한하고, 코어 경로에는 (1) 로그 레벨을 error로 상향하거나, (2) 헬스/카운터 관측성 포인트를 요구하거나, (3) 실패 시 호출자가 알 수 있는 상태 신호를 남기도록 지적한다."
  - `agents/dev-lead.md` — 계획 규율에 항목 추가: "코어 출력 경로(성향·분석 집계 등)에 best-effort try/except를 도입할 때는 실패 가시화 방안(에러 로그 레벨·카운터·헬스 엔드포인트 등)을 계획에 명시하고, 검증 책임자를 할당한다."
  - `CLAUDE.md` (완료 기준) — "완료 기준(Definition of Done)" 체크리스트에 항목 추가: "코어 데이터 생성 경로에 신규 try/except가 추가된 경우, (a) 실패 로그 레벨이 적절한가(warning vs error), (b) 실패가 제품 출력(UI)에 침묵으로 영향을 주지 않는가를 code-reviewer 게이트에서 확인한다."

### B. 높음 — 계약 드리프트 재발: casing 넘어 "필수 필드 존재" 수준

- **현상:** ai_service `/profile/compute`의 move 모델은 `color`(필수)를 요구하나, backend `MoveRecord` 모델(`models/game.py` 확인)에 `color` 필드가 없다. `_collect_moves_by_game`에서 `model_dump`된 dict를 그대로 전달하면 422가 난다. 수정(`profile.py` L156–162)에서 `ply % 2` 패리티로 color를 주입했다. 이전 보고서(2026-06-05)에서 casing 드리프트를 지적했으나, 이번 사이클에는 **필드 자체 존재 여부** 수준으로 재발했다.
- **영향:** 계약 검사가 casing에만 집중하면 "필드 없음"류 드리프트를 잡지 못한다. boundary에서 `MoveRecord`를 dump해 ai_service에 전달하는 경로에 ai_service 스키마와의 대조가 없었다. 이 패턴은 새 ai_service 엔드포인트가 추가될 때마다 재발 가능하다.
- **권장 업데이트:**
  - `agents/code-reviewer.md` — 기존 계약 드리프트 항목을 확장: "서비스 경계(backend→ai_service, FE→backend) 변경 시 양쪽의 required 필드 목록을 대조한다. casing(camelCase/snake_case)뿐 아니라 **한쪽에만 있는 필수 필드(존재 여부)**도 점검한다. dump → 직접 전달 패턴에서 필드 누락은 422로 조용히 실패하므로 코드 정적 확인만으로는 탐지가 어렵다 — 실계약 fixture 또는 통합 호출 시나리오 존재 여부를 확인한다."
  - `agents/implementer.md` — 자가 점검에 항목 추가: "Pydantic 모델을 dump해 다른 서비스 엔드포인트에 직접 전달하는 경우, 상대 서비스의 required 필드 목록과 대조했는가? 특히 새 필수 필드가 상대 스키마에 추가됐을 때 dump된 dict에 해당 필드가 포함되는가."

### C. 중간 — Cloud Run `--set-env-vars`가 env 전체를 치환해 비밀값 유실

- **현상:** backend Cloud Run 재배포 시 `--set-env-vars`를 사용하면 해당 배포에 명시된 변수만 살아남고 이전에 Cloud Run Console에서 수동 설정한 `OPENAI_API_KEY` 등 나머지 env vars가 사라진다(gcloud 동작 특성). 명시적인 가이드가 없었고 배포 후 키 유실로 AI 설명 기능이 동작하지 않는 문제가 발생했다.
- **영향:** 비밀값을 레포에 커밋하지 않고 Cloud Run env에 주입하는 패턴(팀의 올바른 패턴)이 재배포 때마다 유실 위험에 노출된다. `--update-env-vars`(기존 유지 + 신규 추가) 사용법을 모르면 매 배포마다 수동 복구가 필요하다.
- **권장 업데이트:**
  - `agents/migration-engineer.md` — Cloud Run 배포 규칙에 항목 추가: "Cloud Run `--set-env-vars`는 env 전체를 치환한다(이전 env vars 소멸). 기존 변수를 유지하면서 일부만 추가·변경할 때는 `--update-env-vars`를 사용한다. OPENAI_API_KEY 등 수동 주입 비밀값이 있는 서비스는 재배포 전 현재 env를 확인하고(`gcloud run services describe`), 누락될 값을 `--update-env-vars`에 명시하거나 Secret Manager로 이전을 검토한다."

### D. 중간 — gcloud 전역 config 타 프로젝트 오염 — 배포 플래그 필수화 미명문화

- **현상:** 개발 머신의 gcloud 전역 config가 다른 프로젝트(xpp-fitmate)를 가리키는 상태였다. `--project xpp-chess --account teacher@logncoding.com` 플래그를 매번 명시해야 올바른 프로젝트에 배포된다. 이 사실이 팀 지침에 명문화되어 있지 않아 세션마다 사용자가 재확인해야 했다.
- **영향:** 플래그 누락 시 의도하지 않은 프로젝트에 배포가 발생할 수 있다. 특히 CI 없이 로컬 gcloud 로 운영 배포를 하는 현재 구조에서는 인적 실수가 곧 운영 장애로 직결된다.
- **권장 업데이트:**
  - `agents/migration-engineer.md` — 배포 전 체크리스트에 항목 추가: "gcloud를 사용하는 배포 명령은 전역 config에 의존하지 않고 **항상** `--project <project-id> --account <email>` 플래그를 명시한다. 배포 전 `gcloud config list`로 현재 프로젝트·계정을 확인하는 습관을 가진다."
  - `agents/dev-lead.md` — 통합 계획에 항목 추가: "운영 배포가 필요한 작업에서는 배포 명령의 `--project`/`--account` 명시를 계획에 포함하고, 누락 시 implementer/migration-engineer가 멈추고 보고하도록 지시한다."

### E. 중간 — 분석→성향→대시보드 풀 파이프라인 통합 검증 부재

- **현상:** 이번 사이클 최대 발견인 성향 파이프라인 비동작은 단위 테스트·정적 분석으로는 탐지 불가능했다. `trigger_recompute_after_analysis`는 best-effort로 삼켰고, ai_service 호출은 mock으로만 검증됐으며, 실계약 fixture가 없었다. `ImportedGamesAnalyzer.tsx`(파일 확인)는 `sync+fast` 분석 후 `onComplete`로 대시보드를 새로고침하는 풀 플로우를 FE에서 구현했으나, 이 플로우가 실제로 tendency를 산출하는지를 확인하는 통합 테스트가 없다.
- **영향:** "분석 완료 → tendency 생성 → 대시보드 표시"라는 제품의 핵심 가치 흐름이 통합 수준에서 검증된 적이 없다. 앞으로 파이프라인 어느 단계에서든 계약 변경이 생기면 동일한 패턴으로 재발할 수 있다.
- **권장 업데이트:**
  - `agents/dev-lead.md` — 계획 규율에 항목 추가: "분석→성향→대시보드와 같은 멀티-서비스 파이프라인을 포함하는 기능은 QA 핸드오프 기준에 '풀 파이프라인 통합 시나리오(실 ai_service 호출 또는 실계약 fixture)'를 명시한다. mock만으로 파이프라인을 검증했다면 QA 핸드오프 전에 이 사실을 보고한다."
  - `CLAUDE.md` (완료 기준) — 항목 추가: "멀티-서비스 파이프라인(분석·성향·대시보드 등)을 변경한 경우, 실계약 fixture 또는 실 통합 호출로 end-to-end 흐름을 검증했는가를 완료 기준에 포함한다. mock 단독 검증은 완료 조건을 충족하지 않는다."

### F. 중간 — 비밀값 채팅 노출 반복: 구조적 예방책 미비

- **현상:** 사용자가 OPENAI_API_KEY를 2회에 걸쳐 채팅에 평문 붙여넣었다. 팀 가이드는 "키는 레포 금지 → Cloud Run env 또는 Secret Manager"로 올바르게 안내(`config.py` L33–37, `explainer.py` L3)했으나, 실제 사용자는 가이드를 건너뛰고 채팅에 직접 공유했다. 노출 시마다 rotate 권고를 했으나 구조적 예방이 없었다.
- **영향:** rotate를 권고했더라도 노출 자체가 반복되면 키가 로그에 남을 수 있다. 특히 외부 API 키는 노출 즉시 소비·악용될 수 있다. "팀 가이드가 있다"는 것만으로 충분하지 않고, 에이전트가 적극적으로 입력을 요청하지 않고 키 전달 방법을 구체적으로 안내해야 한다.
- **권장 업데이트:**
  - `agents/implementer.md` — 운영 인프라 작업 지침에 항목 추가: "비밀값(API 키, 토큰, 비밀번호)이 필요한 작업에서 사용자에게 값을 요청할 때는 채팅이 아니라 `gcloud run services update --update-env-vars KEY=$(gcloud secrets versions access latest --secret=<name>)` 또는 Cloud Run Console 직접 입력 방법을 구체적으로 안내한다. 채팅/프롬프트에 비밀값을 붙여넣도록 유도하는 질문은 하지 않는다."
  - `agents/dev-lead.md` — 계획 규율에 항목 추가: "비밀값이 필요한 기능(외부 API 연동 등)을 계획할 때, 키 주입 방법(Secret Manager·env var)과 절차를 계획 산출물에 명시한다. 팀원이 키를 채팅에 공유하는 상황을 막으려면 '어떻게 설정할지'를 실행 가능한 명령으로 미리 제공한다."

### G. 낮음 — Cloud Run 인프라 기본값(타임아웃·CPU throttling)의 장기 잡 발목 미문서화

- **현상:** ai_service Cloud Run의 기본 요청 타임아웃이 60초였고, 응답 후 백그라운드 잡에 CPU throttling이 적용돼 60~130수 게임 분석이 완료되지 않았다. 해결 방법은 (1) ai_service `depth=8` 빠른 엔진, (2) backend `/analysis/game` `sync+fast` 옵션, (3) `--no-cpu-throttling`, (4) ai_service 타임아웃 60→300초, (5) backend `analysis_http_timeout=280.0`(`config.py` L29 확인)이었다. 이 지식이 어느 팀 문서에도 없었고 운영 장애 진단 과정에서야 발견됐다.
- **영향:** 장기 잡(분석·마이그레이션·배치)을 Cloud Run에 배포할 때마다 동일한 함정에 빠질 수 있다. migration-engineer가 인프라 변경을 설계할 때 이 기본값을 모르면 설계에 포함하지 않는다.
- **권장 업데이트:**
  - `agents/migration-engineer.md` — Cloud Run 배포 주의사항에 항목 추가: "장기 잡(분석·배치·마이그레이션)을 Cloud Run에 배포할 때 확인할 기본값: (1) 요청 타임아웃 기본 60s → 긴 잡은 `--timeout=300`~`--timeout=3600`으로 상향. (2) CPU throttling 기본값 — 응답 전송 후 CPU가 줄어 백그라운드 처리가 멈출 수 있음. 백그라운드 잡을 요청 내에서 완주해야 하면 `--no-cpu-throttling`. (3) HTTP 클라이언트 타임아웃도 Cloud Run 타임아웃 이하로 맞춘다(현재 `analysis_http_timeout=280`)."
