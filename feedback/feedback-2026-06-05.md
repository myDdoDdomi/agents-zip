## 2. 이슈 상세 (현상 -> 영향 -> 권장 업데이트)

### A. 높음 — 병렬 implementer의 공유파일 경합 및 오너십 위반

- **현상:** 6개 implementer 병렬 수정 세션에서 각 에이전트가 빌드 통과를 위해 할당되지 않은 공유파일(`web/src/services/repositories/types.ts`, `mock.ts`, `firebase.ts`, `eslint.config.js`, `AppModal.vue`)을 "side fix"로 동시 편집. 그 결과 mock 레이어의 `listFollowers`, `suggestFriends`, `suggestGyms` 구현이 단순 스텁으로 덮여 회귀 발생. 그룹장의 결합검증(tsc + lint + build) 루틴에서 사후 발견, 수동 복원.
- **영향:** 이번 세션에서는 그룹장이 포착·복원했으나, 복잡도가 높아지면 무음 회귀로 프로덕션에 유입될 수 있음. 특히 mock 레이어 오염은 테스트 신뢰도를 떨어뜨려 QA 단계에서도 놓칠 위험이 있음.
- **권장 업데이트:**
  - `agents/implementer.md`: "할당된 파일 목록 외 수정 금지. 공유파일(types, mock, firebase, eslint config, 공유 UI 컴포넌트)이 장애물이 되면 임의 수정 대신 멈추고 `(확인 필요)`로 그룹장에 보고" 규칙 명문화.
  - `agents/dev-lead.md`: 병렬 작업 설계 시 "공유파일 단일 오너 지정 + 순차 처리, 또는 병렬 세션 시작 전 공유파일 변경 범위를 명시적으로 락" 규칙 추가. 결합검증(tsc + lint + test + build) 루틴은 이미 효과적이므로 유지.

### B. 높음 — 공유 컴포넌트 전체 재작성 시 기존 동작 유실(회귀)

- **현상:** 재테마 작업 중 `design-system-architect`가 `AppModal.vue`를 전체 재작성하면서, 직전 사용자 요청으로 구현된 반응형 시트 로직(PC = 중앙 다이얼로그 / 모바일 = 바텀시트, `effectiveVariant`/`matchMedia` 기반)을 드롭. 그룹장이 발견하고 수동 복원.
- **영향:** 사용자가 명시적으로 요청한 동작이 설계 의도 없이 소실됨. 재발 시 사용자 요청-구현 사이의 신뢰 훼손. 복원에 추가 컨텍스트 비용 발생.
- **권장 업데이트:**
  - `agents/design-system-architect.md` 및 design 팀 에이전트 전반: "공유 컴포넌트 수정 시 전체 재작성 금지 — diff-aware 최소 변경 원칙. 변경 전 기존 prop/emit/slot 인터페이스와 주요 로직(반응형 분기, matchMedia 등)을 먼저 목록화하고 보존 체크 후 작성." 규칙 추가.
  - 추가로: 기존 컴포넌트를 재작성할 때는 "현재 구현에서 무엇을 유지해야 하는가"를 1~3줄로 명시하도록 작업 시작 템플릿에 포함.

### C. 중간 — FE-백엔드 계약 드리프트의 늦은 발견

- **현상:** FE 레포를 백엔드보다 먼저 구축하면서 콜러블 함수 이름(`rerollMission`, `toggleKudos`, `followUser`, `joinGroupByInviteCode`), 페이로드 키, 반환 shape(`{ok, ...}` 래핑 여부)에 불일치 다수 발생. `code-reviewer` 게이트에서 `createProfile` 역할 규칙 거부(P1 보안), `challenges` 복합 인덱스 누락 등이 추가 발견됨. Cloud Functions `index.js` 전면 작성과 `firestore.rules`·`firestore.indexes.json` 대규모 수정(각각 +181줄, +22줄)이 이 단계에서 집중됨.
- **영향:** 계약 불일치를 늦게 발견할수록 수정 범위가 커짐(FE와 백엔드 양쪽 동시 수정 필요). 이번에는 code-reviewer가 P1 보안 이슈(rules↔쿼리 불일치)를 최후 게이트에서 잡아내, "code-reviewer 안전망 의존" 패턴이 반복됨.
- **권장 업데이트:**
  - `agents/implementer.md`: 백엔드 구현 시작 전 "FE 계약 맵" 산출 단계 필수화. Self-check 항목에 (1) 콜러블 이름·페이로드·반환 shape FE와 대조, (2) 보안 rules에서 list/get 쿼리가 통과하는지 클라이언트 쿼리 기준으로 점검, (3) 복합 인덱스 필요 여부 추가.

## 3. 잘 작동한 점 (유지 권장)

- **토큰 이름 보존 + 값만 교체 재테마 전략:** Bold Outline -> Warm Instrument 전환 시 CSS 변수 이름은 유지하고 값만 교체해 16화면 전체에 재테마가 깔끔하게 전파됨. `design-system-architect`의 이 접근이 재테마 비용을 대폭 절감. 변경 불필요, 유지.
- **그룹장의 병렬 후 결합검증(tsc + lint + test + build) 루틴:** 6개 implementer 병렬 세션 완료 후 tsc·lint·build를 일괄 돌려 공유파일 경합 회귀와 타입 불일치를 일관되게 포착. 이 루틴이 없었으면 회귀가 무음으로 누적됐을 것. `agents/dev-lead.md`에 이미 반영된 것으로 보이나, 필수 단계로 명문화 유지.
- **배포 사람 승인 게이트 준수:** 에뮬레이터 + 단위테스트 65개 통과까지만 진행하고 Firebase 프로덕션 배포는 사람 승인 대기 상태 유지. Human-in-the-loop 원칙 준수. 변경 불필요.
- **QA 4종 병렬 + bug-triager 종합 흐름:** 기능/디자인/접근성/탐색 QA를 병렬로 돌리고 bug-triager가 중복 제거·우선순위 정합을 종합한 흐름이 효율적으로 작동. 이 패턴 유지.
- **백엔드 단위테스트 구조화(functions/lib + functions/test):** `xp.js`, `streak.js`, `weeklyReport.js` 등 비즈니스 로직을 `lib/`에 분리하고 `test/`에 node:test 기반 단위테스트 대응 배치. 65개 테스트 통과 근거 마련. 유지.
- **reference-researcher 선행 리서치 -> 인사이트 기반 결정 패턴:** DESIGN_THEMES.md에서 Strava·WHOOP·Linear 등 14개 출처를 수집, 3개 테마 옵션으로 정리해 사용자 선택을 유도한 흐름이 최종 테마 확정에 효과적이었음. 이 패턴("리서치 -> 옵션 제시 -> 확정 -> 구현")은 이슈 D의 해결책으로도 적용 가능하므로 design-lead 정의에 명문화 권장.
