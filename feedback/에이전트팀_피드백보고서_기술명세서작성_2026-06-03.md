## 2. 이슈 상세 (현상 → 영향 → 권장 업데이트)

### A. 🔴 Sheets 서식의 실제 도구 한계 + 우회 절차 미문서화
- **현상:** 우리 팀 google-workspace MCP는 Sheets에 대해 **`import_to_google_sheets`(값 주입)만** 제공. 셀 배경색·줄바꿈·정렬·열너비·**틀고정·다중 탭(시트)·데이터 유효성(드롭다운)·조건부 서식**을 직접 줄 수 있는 API(batchUpdate/format)가 **없음.** 그런데 `CLAUDE.md §5`는 "Sheets는 읽기뿐 아니라 쓰기/수정도 지원… 직접 작성·수정한다"고만 적혀 있어, 실제로는 **서식 없는 값만** 쓸 수 있다는 한계가 드러나지 않음.
- **영향:** 사용자가 "줄바꿈·배경색·탭"을 요구했을 때 Sheets로는 불가능하다고 잘못 판단하거나, Docs로 우회했다가 다시 Sheets로 되돌리는 등 **수차례 재작업.**
- **검증된 우회법(반드시 문서화):**
  1. 로컬에서 **openpyxl로 .xlsx 생성** — 셀 채우기/폰트/정렬/줄바꿈/열너비/`freeze_panes`/**다중 시트(탭)**/`DataValidation`(드롭다운)/`conditional_formatting`(FormulaRule) 모두 지정.
  2. **`import_to_google_sheets(file_path=file://…, source_format="xlsx")`** 로 업로드 → 변환 시 위 서식·탭·드롭다운·조건부서식이 **그대로 보존**됨(실측 확인).
- **권장 업데이트:** `docs/DOC-FORMATTING.md`에 "**Sheets 서식 = XLSX 빌드 후 import** 표준 절차" 섹션 신설(브랜드색·헤더·틀고정·드롭다운 예시 코드 포함). MCP 가이드에 "Sheets 직접 서식 API 없음" 명시.

### B. 🔴 `doc-formatter`의 오판·중단·토큰 낭비
- **현상:** Sheets 서식 작업을 위임받은 `doc-formatter`가 ① 요청과 달리 **빈 Google Docs를 생성**, ② "**새 스프레드시트를 생성하는 API가 없다**"고 단정하며 사실상 중단, ③ 마크다운 표 복사 안내로 대체. 1회 호출에 **약 129k 토큰** 소모하고 산출물은 폐기됨.
- **영향:** 큰 토큰 낭비 + 그룹장이 직접 XLSX 빌드로 재수행.
- **권장 업데이트(`doc-formatter` 정의):**
  - "**Sheets 산출물 서식이 필요하면**: 마크다운 붙여넣기/Docs 우회로 도피하지 말고, **openpyxl XLSX 빌드 → `import_to_google_sheets` 경로**를 사용한다"를 표준 행동으로 명시.
  - "**새 스프레드시트는 `import_to_google_sheets`로 생성 가능**"(불가하다고 단정 금지).
  - 실패/불확실 시 **중단·임의 대체 금지**, 그룹장에 보고하고 대안 제시.

### C. 🔴 산출 매체(Docs vs Sheets) 사전 확정 절차 부재
- **현상:** '기술 명세서'를 Docs로 만들지 Sheets로 만들지 **착수 전 확정하지 않음.** 가독성 이슈로 Docs로 전환했다가, 사용자가 "기술명세서는 스프레드시트여야 한다"고 재지정 → **Sheet→Docs→Sheet** 왕복.
- **영향:** 동일 산출물 매체 재작업 2회 이상.
- **권장 업데이트(`CLAUDE.md §5` + 라우팅):**
  - "**상위 문서가 있으면 그 매체를 따른다**" 규칙을 더 강하게(요구사항정의서가 Sheets였으므로 기술명세서도 Sheets가 기본).
  - 착수 전 **매체 1줄 확인**을 표준 게이트로(특히 표 중심 문서).
  - §5의 Sheets-권장 목록에 **기술명세서** 명시 추가.

### D. 🟠 '기술 명세서' 전용 템플릿·스킬 부재
- **현상:** 라우팅 표에 **기능명세서(`/functional-spec`)** 는 있으나 **기술 명세서**는 없음. 9컬럼(요구사항정의서 양식을 기술 재해석: ID·모듈·기능명·사용자가치·**기술구현방식·기술비기능·처리흐름·예외**·Phase·우선순위) 양식을 **즉석 정의**해 진행.
- **권장 업데이트:**
  - `templates/기술명세서.md` 신설(위 9컬럼 + 작성 가이드 + 셀 표기 규칙).
  - `.claude/skills/`에 `/tech-spec` 스킬 추가, 담당 `spec-writer` 매핑.
  - 라우팅 표에 행 추가.

### E. 🟠 import 운영 함정 (경로·파일명)
- **현상:**
  1. `import_to_google_sheets`의 `file_path`는 **`C:\Users\info\.workspace-mcp\attachments\` 내부 파일만 허용** + **`file://` URL** 필요(절대 Windows 경로는 거부) → 2회 실패 후 파악.
  2. `file_name`의 **마지막 점 뒤가 확장자로 잘림**(`동행AI_…_v3.1` → `v3`). → 점 없는 이름으로 import 후 `update_drive_file(name=…)`로 리네임 필요.
- **권장 업데이트:** MCP 가이드에 두 함정과 회피법(허용 디렉터리·file:// 사용, 점 없는 임시명→리네임) 명시.

### F. 🟡 휴지통 정리 권한 + 임시파일 양산
- **현상:** 재생성 과정에서 **임시/오생성 Drive 파일 5~6개** 발생. `update_drive_file(trashed=true)` 정리가 **자동 분류기에 의해 1회 차단**(사용자가 명시 삭제 요청 안 한 공유파일).
- **권장 업데이트:**
  - **"한 번에 올바르게" 원칙** 강화 — 매체·서식 확정 후 단일 산출(중간본 양산 금지, B-3 "추측 ID 병렬 호출 금지"와 동일 취지).
  - 정리가 필요한 경우 **그룹장이 사용자 확인 후 일괄 정리**.

### G. 🟠 대량 콘텐츠 재전송 반복 (토큰 효율)
- **현상:** 동일한 34행 콘텐츠를 **모듈 작성→검수→재서식**마다 에이전트 프롬프트로 재전송. spec-writer를 모듈마다 새로 spawn하며 풀 브리프 반복.
- **권장 업데이트(`CLAUDE.md §2-1` 효율 수칙):**
  - **중간 산출물을 먼저 Drive/로컬에 저장** → 후속 에이전트(검수·서식)는 **파일을 읽게** 하여 인라인 재전송 제거(`get_drive_file_content`).
  - 동일 작업 연속은 **새 spawn 대신 같은 에이전트 이어쓰기(SendMessage)** 로 컨텍스트 재사용.

## 3. 잘 작동한 점 (유지 권장)
- **`spec-writer`**: 모듈별 9컬럼 명세를 일관 양식·적정 깊이로 산출. 백엔드 설계 가이드를 정확히 반영(환각방지 2단계 가드, 캐시 키 양자화, 비용 카운트 트랜잭션 등). 품질 우수.
- **`doc-reviewer`**: 통합 검수에서 **끊긴 REQ 상호참조·용어 2축 혼용·검증불가 임계치**를 정확히 적발하고 심각도·수정안 제시("조건부 통과" 판정). 게이트 역할 충실 — 변경 불필요.
- **거버넌스**: `(확인 필요)` 표기로 추측 배제, 결정사항 책임자 분류 — 유효하게 작동.
