## 2. 이슈 상세 (현상 -> 영향 -> 권장 업데이트)

### A. 높음 — 캔버스/브라우저 전용 컴포넌트 목킹이 실브라우저 렌더 결함을 은폐

- **현상:** chessground(체스 보드 라이브러리)는 실제 DOM 조작·캔버스 렌더를 사용해 JSDOM에서 동작하지 않는다. Board 단위테스트(`Board.test.tsx`)는 `getBoundingClientRect`를 stub하고 "예외 없이 마운트된다"만 검증한다. 이 테스트는 650개 전체 green 상태를 유지했으나, 실사용자가 보는 보드 미표시 결함(흰 화면)은 단위테스트로 잡을 수 없었다. 실원인 확인은 실브라우저 E2E(Playwright, `qa-smoke.mjs`) 도입 이후에야 가능했다.
- **영향:** 단위테스트 그린 = 보드 정상 렌더가 아님. 팀 전체가 "테스트 통과 = 안전"으로 오해하면, 목킹된 캔버스/실시간/WebSocket 영역의 시각 결함이 릴리스까지 도달한다. Sev1/2 결함이 게이트를 통과하는 구조적 위험.
- **권장 업데이트:**
  - `agents/code-reviewer.md` — 리뷰 관점 #5(컨벤션·테스트)에 항목 추가: "chessground·canvas·WebGL·WebSocket 등 JSDOM에서 목킹되는 브라우저 전용 컴포넌트를 포함하는 변경에는 단위테스트만으로 불충분하며, 실브라우저 E2E 검증 계획이 있는지 확인한다."
  - `CLAUDE.md` (완료 기준, `dev_agents` 원본 레포) — "완료 기준(Definition of Done)" 체크리스트에 항목 추가: "브라우저 전용 렌더(canvas/WebGL/chessground 계열) 변경 시 실브라우저 E2E(Playwright 등) 게이트 필수 — 단위테스트 목킹만으로 완료 불가."
  - `agents/implementer.md` — 자가 점검 항목 추가: "캔버스·브라우저 전용 컴포넌트(chessground 등)를 변경했다면 단위테스트 목킹이 실결함을 은폐하지 않는가? 실브라우저 E2E 게이트를 QA 핸드오프로 명시했는가."

### B. 중간 — 인프라/네트워크 상태 우선 점검 없이 코드 결함으로 오귀인

- **현상:** 사용자가 보고한 "빈 화면"을 코드 결함(보드 렌더링 버그)으로 오래 추적했으나, 실제 원인은 uvicorn --reload 백엔드 프로세스 비정상 종료 + CORS 차단으로 인한 데이터 미로딩이었다. 프로세스 상태·콘솔 네트워크 오류·CORS 헤더를 먼저 확인했다면 진단 시간을 크게 줄일 수 있었다.
- **영향:** 인프라 상태를 코드 레벨에서 추적하면 불필요한 코드 변경·재검증 루프가 발생한다. 특히 uvicorn --reload가 타 서비스 재시작 시 깨지는 패턴은 이 프로젝트 고유 운영 지식으로, 반복 발생 가능성이 높다.
- **권장 업데이트:**
  - `agents/debugger.md` — 작업 워크플로우 1단계(재현) 앞에 "인프라/환경 상태 선점검" 체크리스트 추가: "(1) 브라우저 콘솔 네트워크 탭 — 4xx/5xx·CORS 오류 여부, (2) 백엔드 프로세스 생존 여부(포트 응답·로그), (3) CORS 설정 일치 여부 — 이 세 가지를 코드 레벨 추적 전에 먼저 확인한다. 흰 화면/빈 데이터는 네트워크 단절이 코드 결함보다 흔하다."

### C. 중간 — 설계 SSOT 도달성 점검 누락으로 P0 요구사항이 implementer 단계를 통과

- **현상:** 죽은 라우트 `/pvp/leaderboard`, 온보딩 SCR-27(고아, REQ-60 P0), 홈 화면 Streak 미배치(REQ-62 P0)가 implementer 구현 후 code-reviewer/design-reviewer 단계에서야 발견됐다. implementer self-check에 "구현 완료 화면이 설계 SCR 번호와 대응하는가, 네비게이션으로 도달 가능한가"를 묻는 항목이 없었다.
- **영향:** P0 요구사항 누락이 앞단에서 막히지 않으면 code-reviewer·design-reviewer가 "설계 준수 감사" 부담을 지게 되어 리뷰 비용이 증가하고, dev-lead 계획에도 빈틈이 생긴다.
- **권장 업데이트:**
  - `agents/implementer.md` — 자가 점검에 항목 추가: "(설계 문서 기반 구현 시) 구현한 화면/기능이 설계 SCR 번호에 대응하는가? 해당 화면이 네비게이션(Sidebar·라우트)으로 실제 도달 가능한가? P0 요구사항이 이번 범위에 포함된 경우 누락 없이 구현했는가."
  - `agents/dev-lead.md` — 계획 규율에 항목 추가: "P0/P1 요구사항(REQ 번호)을 포함하는 작업 분해에서는 각 항목이 어느 화면(SCR)에 배치되는지 표에 명시하고, implementer self-check 또는 design-reviewer 게이트에서 닫히는지 확인한다."

### D. 중간 — 신규/빈 사용자 404 처리 self-check 항목 부재

- **현상:** 신규 가입 사용자의 `users/{uid}` 문서가 아직 생성되기 전인 경우 Firestore 404를 반환하는데, 프론트엔드 여러 컴포넌트가 이를 오류로 처리하거나 빈 상태를 렌더링하지 못했다. `/users/ensure` 호출 누락이 반복됐고, 매 구현마다 수동으로 발견됐다.
- **영향:** 신규 사용자 온보딩 경험이 반복적으로 깨진다. Firestore의 "문서 없음 = 빈 상태(not error)" 패턴은 이 프로젝트 전반에 걸친 관례이므로, 모든 구현자가 인지해야 한다.
- **권장 업데이트:**
  - `agents/implementer.md` — 자가 점검에 항목 추가: "신규 사용자 또는 데이터 없는 상태(404·빈 컬렉션)를 처리하는가? Firestore의 문서 없음은 오류가 아닌 빈 상태로 처리한다. users/{uid}에 의존하는 기능은 /users/ensure 선행 호출 또는 빈 상태 폴백을 구현했는가."

### E. 낮음 — 디자인 토큰 정의-사용 일치 미검증으로 Tailwind 무음 드롭 발생

- **현상:** `rounded-pill` 유틸이 35곳에 사용됐으나 `tailwind.config.js`에 정의되지 않아 Tailwind가 조용히 버렸다. `--badge-fg`, `--board-*` CSS 변수도 컴포넌트에서 참조되나 `:root`/`.dark`에 정의가 없었다. 단위테스트와 빌드 모두 오류 없이 통과해 시각 결함이 뒤늦게 발견됐다.
- **영향:** 디자인 시스템 적용 시 토큰 추가가 정의(tailwind.config.js/index.css)와 사용(컴포넌트)이 분리된 두 곳에서 이뤄지는데, 한쪽 누락이 빌드 오류로 드러나지 않아 시각 회귀가 자동 탐지되지 않는다.
- **권장 업데이트:**
  - `agents/code-reviewer.md` — 리뷰 관점 #3(구조·유지보수성)에 항목 추가: "새 Tailwind 유틸 클래스(`rounded-*`, `text-*`, `bg-*` 등) 또는 CSS 변수(`--color-*`, `--badge-*` 등)를 사용하는 경우, `tailwind.config.js` / `index.css`에 정의가 존재하는지 확인한다. Tailwind는 미정의 유틸을 오류 없이 무시하므로 빌드 성공이 정의 존재를 보장하지 않는다."
  - `agents/implementer.md` — 자가 점검에 항목 추가: "새 Tailwind 클래스나 CSS 변수를 사용한 경우 tailwind.config.js·index.css에 정의를 추가했는가? (정의 없이 사용하면 Tailwind가 조용히 무시해 시각 결함이 발생한다.)"

## 3. 잘 작동한 점 (유지 권장)

- **디자인 -> dev -> QA -> 수정 풀사이클 오케스트레이션:** ui-designer 스펙 산출 후 implementer 구현, QA 팬아웃(accessibility-tester·regression·exploratory·design-reviewer), code-reviewer, dev 수정, 재검증까지 역할 경계를 유지하며 전사이클을 완주했다. 팀 구조가 실전에서 유효함을 검증. 변경 불필요.
- **병렬 implementer 파일 경계 위임 (충돌 0):** 27개 화면의 시맨틱 토큰 마이그레이션을 파일 단위로 분할해 병렬 위임, 병합 충돌 없이 완료. 독립적 작업을 병렬 처리하는 위임 패턴이 효과적으로 작동. 변경 불필요.
- **code-reviewer의 실결함 포착 안전망:** 매 라운드 실결함(stale FEN 기반 사운드 오분류 버그, Escape 키 리로드 부작용, 허위 claimTimeout 트리거)을 정적 추론이 아닌 코드 근거로 포착했다. code-reviewer의 read-only + 표 기반 보고 형식이 리뷰 가독성과 추적성을 높임. 변경 불필요.
- **실브라우저 E2E 도입(channel:msedge, Playwright):** `qa-smoke.mjs`로 전체 라우트 스모크 + `landing-check.mjs`로 랜딩 전용 검증을 구축했다. JSDOM 목킹이 잡을 수 없는 실렌더 결함을 탐지하는 유일한 게이트로 기능. 앞으로 확장 권장. 변경 불필요.
- **메모리에 비자명 운영 지식 기록:** uvicorn --reload 동작 패턴, contract drift 패턴, 디자인 시스템 SSOT 위치 등 재발견 비용이 큰 지식을 `memory/` 파일에 기록해 세션 간 인수인계. 변경 불필요.
