# /update-agents — 본부 자기개선(Self-Update) 파이프라인

IT 전반의 동향을 **균등·광범위**하게 리서치하고, 그게 어느 부서 팀에 어떻게 의미 있는지로
**번역**해, **트렌드 관련 팀만** 골라 디벨롭하는 본부의 자기개선 사이클이다. 이 스킬은
본부 전담 팀원 3명(`trend-researcher`·`update-curator`·`team-fit-reviewer`)을 순서대로/병렬로
부리고, 승인된 제안을 `team-architect`가 적용한다. **트리거는 수동(`/update-agents`)만**이다.

> **빌트인 웹검색만 쓴다(외부 MCP 불요).** 수집은 `trend-researcher`의 `WebSearch`/`WebFetch`로 한다.
> **게이트 4개**(②동향 리포트 · 3-b선별 목록 확인 · ⑥인터뷰 보완 · ⑦적용 승인)를 거친다 — 사람 승인 없이 팀 파일을 고치지 않는다.
> 흐름: 리서치 → **②게이트** → 필터·팀선별 → **3-b게이트** → 팀별 병렬 검토(팬아웃) → 종합 → **⑥게이트** → **⑦게이트** → 적용 → runs 기록.

## 0. 상태 확인 (diff 기준점)

1. `.claude/skills/update-agents/runs/`를 `Glob`(`update-*.md`)으로 본다.
   - `update-YYYY-MM-DD.md` 파일들 중 **날짜 최댓값**이 **diff 기준점**이다(가장 최근 사이클).
   - 파일이 없으면(=`.gitkeep`만) **첫 실행 모드** — 현 지형 전체를 리서치한다.

## 1. 수집 — trend-researcher

2. `trend-researcher`에게 위임한다.
   - **첫 실행**이면 "현 지형 전체"를, **2회차+**면 §0에서 찾은 **기준 날짜**를 전달해 "그 날짜 이후 diff만" 수집하게 한다.
   - 인자(`$ARGUMENTS`)로 관심 영역/주제를 받았으면 함께 전달한다(없으면 IT 전반 균등 커버).
   - 산출: 분야별 **동향 리서치 노트**(각 항목에 출처 URL·날짜·한 줄 의미).

## 2. [게이트 ②] 동향 리포트 제시

3. 사용자에게 리서치 결과를 **분야별 요약**으로 제시한다(가장 주목할 5개 포함).
   - "이대로 디벨롭 검토를 진행할까요? (영역 추가/제외 반영 가능)"을 묻는다.
   - 사용자가 영역을 가감하면 §1을 보강 호출하거나 노트를 조정한 뒤 진행한다. 승인 전에는 다음 단계로 넘어가지 않는다.

## 3. 필터·선별 — update-curator

4. `update-curator`에게 동향 노트를 넘겨 위임한다(웹 재검색 없이 노트만으로 판단).
   - 루트 `CLAUDE.md` §5 팀 레지스트리와 대조해 **선별 항목 × 영향 팀 매핑 + 우선순위**를 만들고,
     **트렌드 관련 팀만** 골라 **선별된 팀 목록**을 산출하게 한다.
   - 산출: **큐레이션 노트**(매핑표 + 선별된 팀 목록 + 탈락 항목).

## 3-b. [게이트 3-b] 선별 목록 확인 (신규)

5. 사용자에게 **"이번엔 <선별된 팀들>을 검토할게요"** 라고 선별된 팀 목록을 제시한다.
   - 사용자가 팀을 **추가/제외** 조정할 수 있다(예: "design은 빼고 marketing 추가"). 조정을 반영한다.
   - 확인된 팀 목록이 다음 단계 팬아웃 대상이다. 확인 전에는 팬아웃하지 않는다.

## 4. 팀별 병렬 검토 — team-fit-reviewer (팬아웃)

> 이 팬아웃은 Claude Code **Dynamic Workflows**(오케스트레이터-워커 병렬 스폰)라는 1급 패턴에 기반한다.

6. **확인된 팀마다 `team-fit-reviewer`를 1회씩 병렬 호출**한다(팬아웃).
   - 각 호출에 **그 팀 1개 + 그 팀에 매핑된 트렌드 항목**(큐레이션 노트에서)을 전달한다.
   - 각 리뷰어는 그 팀 폴더만 정독해 **팀 디벨롭 제안**(제안·근거·예상 변경 파일·우선순위)을 반환한다(read-only, 적용 안 함).

## 5. 종합

7. 팀별 제안을 **본부 관점으로 통합**한다: 중복·충돌 정리, 전체 우선순위 재배열, 공통 패턴(여러 팀에 걸친 표준 갱신)은 묶는다.

## 6. [게이트 ⑥] 인터뷰 보완

8. 종합 결과에서 **근거가 부족하거나 선택이 필요한 항목만** 사용자에게 **1~3문** 인터뷰한다.
   - (예: "이 MCP는 권한 리스크가 있는데 read-only로 제한할까요?", "새 역할을 신설할까요, 기존 역할 보강일까요?")
   - 모든 항목을 묻지 않는다 — 정할 수 있는 건 정하고 기록만 한다(ask-rate 보정).

## 7. [게이트 ⑦] 적용 승인

9. 통합·보완된 제안을 **적용 후보 목록**으로 제시하고 **"이 변경을 적용할까요?"** 를 묻는다(승인 항목만 적용).
   - 팀 파일 수정·파괴적 작업 전 **사람 승인**이 전제다. 승인되지 않은 항목은 다음 사이클로 보류한다.

## 8. 적용 — team-architect

10. 승인된 제안을 `team-architect`에게 전달한다.
    - `team-architect`가 §증분 추가·기존 팀 개선 절차로 팀 파일(에이전트·스킬·CLAUDE.md·템플릿)을 표준·체크리스트에 맞춰 수정한다.
    - 새 팀 신설이 필요하면 `team-architect`의 설계안→승인 절차를 따른다.

## 9. 기록 — runs/

11. 이번 사이클을 `runs/update-YYYY-MM-DD.md`로 저장한다(**오늘 날짜**, 세션 컨텍스트 기준).
    - **같은 날 재실행이면** `update-YYYY-MM-DD-2.md`(`-3`…)로 접미해 직전 기록을 덮어쓰지 않는다(재실행 안전).
    - 기록 내용: ① 동향 리포트 ② 선별·매핑(선별된 팀 목록 포함) ③ 팀별 제안 ④ 적용 로그(무엇을 어느 팀에 반영했나).
    - 이 파일의 날짜가 **다음 실행의 diff 기준점**이 된다.
    - **즉시 재사용 가능한 학습**(반복되는 선별 기준·어떤 트렌드를 어느 팀에 반영했는지)은 루트 `MEMORY.md`에도 반영한다(루프 닫기).

## 10. 보고

12. 사용자에게 요약 보고: 리서치 분야 수, 선별된 팀, 팀별 적용 항목 수, `runs/` 기록 경로, 다음 사이클 diff 기준 날짜.

```markdown
# /update-agents 사이클 기록 — YYYY-MM-DD
> 모드: <첫 실행 / diff(기준 YYYY-MM-DD)> | 관심 영역: <인자 또는 IT 전반 균등>

## 1. 동향 리포트 (요약)
- <분야> : <주목 항목 1줄> (출처)

## 2. 선별·매핑
| 트렌드 항목 | 영향 팀 | 반영 방식 | 우선순위 |
|---|---|---|---|
선별된 팀: <…> / 제외: <…>

## 3. 팀별 제안 (team-fit-reviewer 팬아웃)
### <팀명>
- <제안> — <근거> — <예상 변경 파일> — <우선순위>

## 4. 적용 로그
- [x] <팀> <파일> — <무엇을 반영> (승인일 YYYY-MM-DD)
- [ ] <보류 항목> — 다음 사이클

## 5. 다음 사이클 메모
- diff 기준점: 이 파일 날짜 / MEMORY.md 반영: <있으면 항목>
```

## 주의
- **게이트 4개를 건너뛰지 않는다.** ②·3-b·⑥·⑦ 각각 사용자 승인 후 진행한다(팀 파일 수정은 ⑦ 이후).
- **중복 호출 금지.** trend-researcher는 1회 수집, update-curator는 웹 재검색 없이 그 노트로만 판단.
- **트렌드 관련 팀만** 디벨롭 대상 — 영향 없는 팀은 건드리지 않는다.
- 인자(`$ARGUMENTS`)가 있으면 관심 영역/주제로 사용하고, 없으면 IT 전반을 균등 리서치한다.
