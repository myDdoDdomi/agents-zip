# /team-assemble — 멀티팀 조립 오케스트레이터

여러 부서 에이전트 팀(dev·QA·design·docs·marketing·resume 등)을 **한 프로젝트에서 함께** 쓰고 싶을 때,
각 팀을 `.claude/agents/<팀>/` 하위폴더로 정리해 두면 이 스킬이 그 폴더들을 읽어 **충돌을 해결하고
협업용 `CLAUDE.md`를 자동으로 만들어/갱신**한다. 팀이 늘거나 줄면 다시 실행만 하면 된다(재실행 안전).

## 0. 왜 이 스킬이 필요한가 (Claude Code 동작 근거)

- `.claude/agents/`는 **하위폴더까지 재귀 인식**된다 → 팀별 폴더(`dev/`, `qa/` …)로 정리 가능.
- 그러나 에이전트의 **식별자는 `name` frontmatter뿐**이고 **폴더 경로는 무시**된다. 따라서
  서로 다른 팀이 **같은 `name`**(예: 두 팀이 모두 `code-reviewer`·`a11y-reviewer`를 가질 때)을 가지면
  **경고 없이 하나가 사라진다.** → 이 스킬이 **충돌을 자동 탐지·리네임**해 모든 에이전트가 공존하게 만든다.
- **`feedback-reporter`만은 예외(특수 케이스).** 모든 팀이 같은 이름으로 두는 공통 기능이라 리네임 대상이
  아니라, **단일 통합 reporter로 수렴**시킨다(§2-2-b). reporter가 하나도 안 복사됐으면 공통 위치에 생성한다.
  그리고 이를 호출하는 **통합 `/feedback-agents` 스킬도 이 스킬이 생성/갱신**한다(§2-4c) → 어떻게 복사했든
  `/feedback-agents` 한 번으로 조립된 전 팀을 회고해 본부에 **통합 보고**된다.
- 스킬은 일반 도구(Read/Glob/Edit/Write)로 파일을 읽고 `CLAUDE.md`를 고칠 수 있다(샌드박스 없음).

## 1. 전제 — 폴더 구조

대상 프로젝트(여러 팀을 함께 쓸 곳)에 팀별 하위폴더로 에이전트를 둔다:

```
대상-프로젝트/
└── .claude/
    ├── agents/
    │   ├── dev/        # dev_agents/agents/*.md 를 복사
    │   │   ├── architect.md
    │   │   └── …
    │   ├── qa/         # QA_agents/agents/*.md 를 복사
    │   │   └── …
    │   └── design/     # design_agents/.claude/agents/*.md 를 복사
    │       └── …
    ├── skills/
    │   └── team-assemble/SKILL.md   # (이 스킬)
    └── CLAUDE.md       # ← 이 스킬이 생성/갱신 (없으면 새로 만듦)
```

> 모델 B 팀(dev·QA)은 원래 `agents/`가 평면이고, 모델 A 팀(design·docs·marketing·resume·edu·webnovel)은 `.claude/agents/`에 있다.
> **어느 쪽이든 그 팀의 에이전트 `*.md`들을 `대상/.claude/agents/<팀>/`로 복사**해 두면 된다.

### 1-1. 무엇을 복사하나 — 모델 A vs 모델 B (조립 차이)

모델 A 팀은 에이전트가 자기 팀의 **스킬·템플릿·`.mcp.json`(Drive·Figma 등)** 에 의존한다.
**에이전트만** 옮기면 그 의존이 끊겨 기능이 깨진다 → 모델 A는 **4종을 함께** 옮겨야 자기완결로 동작한다.

| 옮기는 것 | 모델 B (복사-라이브러리: dev·QA) | 모델 A (자기완결형: docs·design·marketing·resume·edu·webnovel) |
|---|---|---|
| ① 에이전트 | `agents/*.md` → `대상/.claude/agents/<팀>/`(평면) | `.claude/agents/*.md` → `대상/.claude/agents/<팀>/` |
| ② 스킬(슬래시 명령) | 도메인 스킬만 `대상/.claude/skills/`로(예: dev의 `/pentest`·`/remediate`) | `.claude/skills/*` → `대상/.claude/skills/`(슬래시 명령 동작) |
| ③ 템플릿 | (없음) | `templates/*` → `대상/templates/`(에이전트가 참조하는 상대경로 유지) |
| ④ MCP 설정 | (없음) | `.mcp.json`이 있으면 **대상 `.mcp.json`에 병합**(비밀값은 `${ENV}` 유지, 키 충돌 시 사용자 확인) |
| ⑤ 참조 문서 | (없음) | 에이전트가 상대경로로 참조하는 `docs/`·`config/` 등 → `대상/docs/`·`대상/config/`에 병합(예: docs 팀의 `DOC-FORMATTING.md`·`STYLE-GUIDE.md`·`drive-config.md`는 에이전트의 절대 기준 — 없으면 기능 저하) |

> **모델 A 포트(이식) 주의:** 모델 A 에이전트 본문이 자기 팀 `CLAUDE.md`(미복사)를 참조하는 부분은
> 조립 환경에서 **협업 `CLAUDE.md`의 AUTO 블록·공통 거버넌스**(§3)로 대체된다. 핵심 산출 규칙·자가 점검은
> 에이전트 본문에 자기완결적으로 들어 있어야 동작한다(앞으로 만들 팀은 `team-architect` §산출물 2가 강제).
> **②~⑤는 §2-1b 자동 포트가 처리한다** — 사용자는 ①에이전트만 복사해 와도 된다(원본 HQ 레포가 같은
> 머신에 있고 경로를 알 때). HQ 레포가 없는 환경이면 위 표대로 수동 복사·병합하고, 스캔 후 보고에서 누락 시 안내한다.
> **`/feedback-agents`는 복사 안 해도 된다 — `/team-assemble`이 통합 본문으로 생성/갱신한다(§2-4c).** ②에서는
> **도메인 스킬만** 옮기면 된다. 통합 reporter도 없으면 공통 위치에 생성되므로 명령이 항상 동작한다(§2-2-b·§2-4c).

### 1-2. 재배포(팀 업데이트) 체크리스트 — 수동 복사 사고 방지

원본 팀 레포에서 **업데이트된 에이전트를 기존 조립 프로젝트에 다시 배포**할 때 아래를 지킨다.
(실사고 2026-06-10: docs 에이전트 16종이 `.claude/agents/dev/`에 오배치돼 dev 에이전트와 이름 전면 충돌 +
협업 `CLAUDE.md`가 원본 팀 CLAUDE.md로 통째 덮어써져 멀티팀 오케스트레이터 블록·타 팀 그룹장 본문이 소실됐다.
CLAUDE.md 덮어쓰기는 팀 운영 규칙 전체를 무력화하는 파괴적 사고다.)

- [ ] **복사 원본은 HQ 레포에서만 가져온다.** 대상 프로젝트 안의 유사 명칭 팀 폴더(예: `dev\design_agents` — 프로젝트 전용 구버전)는 HQ가 아니다 — 구버전으로 재조립하면 기능·정책이 어긋난다(혼동 실사례 2026-06-11).
- [ ] **팀별 폴더에만 복사한다**: `<팀> 에이전트 → .claude/agents/<그 팀>/`. 복사 직후 폴더명과 에이전트 소속이 일치하는지 확인한다(docs 에이전트가 `dev/`에 들어가지 않게).
- [ ] **협업 `CLAUDE.md`는 덮어쓰지 않는다.** 원본 팀 CLAUDE.md 내용이 필요하면 마커 **밖**에 별도 섹션으로 병합하거나 `@import`로 참조한다. AUTO 블록 갱신은 이 스킬 재실행으로만 한다.
- [ ] **동명 파일은 diff 확인 후 교체한다**(신버전인지 확인). 교체되는 구버전은 백업 폴더(예: `.claude/agents_backup_YYYYMMDD/`)에 보존한다.
- [ ] 재배포 후 **이 스킬을 다시 실행**해 충돌 해결·라우팅 표를 재생성하고, 마커 밖의 사용자 작성 내용(팀 그룹장 본문 등)이 보존됐는지 확인한다.

## 2. 절차 (이 순서대로 수행)

### 2-1. 스캔
1. 대상 agents 루트를 정한다: **인자가 있으면 그 경로**, 없으면 `./.claude/agents/`.
2. `Glob`으로 `<루트>/**/*.md` 전부 찾는다.
3. 각 파일의 **팀**을 정한다 = 루트 바로 아래 **첫 폴더명**(예: `…/agents/qa/foo.md` → 팀 `qa`).
   루트 바로 밑(하위폴더 없이) 있는 파일은 팀 `(공통)`으로 둔다.
4. 각 파일을 `Read`해 frontmatter(맨 위 `---`~`---`)에서 `name`·`description`·`model`·`tools`를 추출한다.
   frontmatter가 없거나 `name`이 없으면 **경고 목록**에 적고 건너뛴다(임의 생성 금지).

### 2-1b. 모델 A 동반 자산 자동 포트 (에이전트만 복사해 와도 되게)

> 목적: 사용자가 **①에이전트만** `.claude/agents/<팀>/`로 복사해 와도, 이 스킬이 동반 자산(②스킬·③템플릿·
> ④`.mcp.json` 병합·⑤참조 문서)을 **원본 HQ 레포에서 자동으로 가져와** 모델 A 팀이 자기완결로 동작하게 한다.
> 단, **환경변수 설정·브라우저 OAuth 1회 인증은 자동화 불가**(비밀값·사람 몫) — 잔여 작업으로 보고한다.

4-1. 스캔된 팀 중 **모델 A 팀**(docs·design·marketing·resume·aiot·pm·edu·webnovel — HQ 레지스트리 기준)이
   있으면, 그 팀의 동반 자산이 대상에 이미 있는지 확인한다(그 팀 대표 스킬 폴더·`templates/` 파일 존재 여부).
   전부 있으면 이 단계는 건너뛴다(재실행 안전).
4-2. 누락이면 **원본 HQ 레포 경로**를 확보한다: 협업 `MEMORY.md` §2(환경 매핑)에 기록된 경로를 먼저 찾고,
   없으면 사용자에게 **1회만** 묻고 그 답을 `MEMORY.md` §2에 기록한다(재실행 시 재사용). HQ 레포가 이 머신에
   없으면 자동 포트를 건너뛰고 §1-1 표의 수동 복사를 안내한다.
4-3. HQ 레포에서 대상으로 복사한다(**Read→Write 파일 단위 복사** — 이 스킬의 도구 범위 내, 셸 불사용):
   - **스킬**: `<HQ>/<팀>/.claude/skills/*` → `대상/.claude/skills/` — 단 `feedback-agents`는 복사하지 않는다
     (§2-4c가 통합본을 생성). 동일명 스킬 폴더가 이미 있으면 **덮어쓰지 않고** 보고에 남긴다(diff 확인 권유).
   - **템플릿**: `<HQ>/<팀>/templates/*` → `대상/templates/` — 동일 파일명이 있으면 덮어쓰지 않고 보고.
   - **참조 문서**: 그 팀 폴더의 `docs/`·`config/` 등 에이전트·스킬이 상대경로로 참조하는 폴더를
     `대상/docs/`·`대상/config/`에 **병합**한다(파일명 충돌 시 덮어쓰지 않고 보고 — 여러 팀의 `docs/`는
     파일명이 다르면 한 폴더에 공존 가능).
   - **모델 B 팀(dev·QA)**: 도메인 스킬만(예: `/pentest`·`/remediate`) 복사 — 그 외 동반 자산은 없다.
4-4. **`.mcp.json` 병합**: `<HQ>/<팀>/.mcp.json`의 `mcpServers` 항목들을 대상 `.mcp.json`에 병합한다(없으면 생성).
   - 같은 서버 키가 이미 있고 내용이 다르면 **덮어쓰지 않고 사용자 확인**.
   - `${ENV}` 치환은 그대로 유지한다. **실값 비밀(토큰·secret)이 보이면 그 항목 복사를 멈추고 보고**한다(비밀값 레포 금지).
   - `$comment`·`$<key>_optional` 류 비활성 블록은 함께 옮긴다(활성화는 사용자 판단).
   - **편집 게이트 주의:** Claude Code 권한 게이트가 `.mcp.json` 생성·수정을 차단할 수 있다(실사례
     2026-06-11, 2회). 차단되면 우회하지 말고, 병합 결과를 **완성된 JSON 블록으로 보고에 제시**해
     사용자가 직접 붙여넣게 한다(이 경우 ENV·OAuth와 함께 잔여 수동 작업으로 분류).
4-5. **잔여 수동 작업을 보고에 명시**: 필요한 환경변수 목록(예: docs 팀 `GOOGLE_CLIENT_SECRET_PATH` 등,
   pm 팀 `SLACK_CLIENT_ID`/`SLACK_CLIENT_SECRET`)과 브라우저 OAuth 1회 인증, 그리고 가이드 문서 경로
   (함께 복사된 `docs/*-MCP.md`)를 안내한다.

### 2-2. 이름 충돌 해결 (필수 — 자동)

**(a) 일반 충돌 — 팀 접두어 리네임**
5. 모든 파일의 `name`을 모아 **2개 이상 파일에서 겹치는 이름**을 찾는다. (단 `feedback-reporter`는 (b)로 특수처리 — 제외)
6. 겹치는 각 이름에 대해, **충돌한 파일 전부**의 `name`을 `<팀슬러그>-<원래이름>`으로 바꾼다.
   - 예: 두 팀이 모두 `code-reviewer`를 가지면 `dev/code-reviewer.md` → `name: dev-code-reviewer`,
     `qa/code-reviewer.md` → `name: qa-code-reviewer`.
   - `Edit`로 **frontmatter의 `name:` 줄만** 교체한다(본문·다른 줄은 건드리지 않음).
   - 리네임 후에도 겹치면(드묾) 끝에 `-2`,`-3`… 붙인다.
   - **본문은 자동 수정하지 않는다.** 단, 리네임된 이름이 다른 에이전트 본문에서 호출될 수 있으니
     **리네임 목록을 리포트에 남겨** 사용자가 확인하게 한다.
   - 충돌이 없으면 이 단계는 건너뛴다(원래 이름 유지).
   > 기본 정책은 **"충돌 시에만 리네임"**(원래 이름 최대 보존). 사용자가 "전부 팀 접두어로 통일"을 원하면
   > 모든 에이전트를 `<팀>-<이름>`으로 바꾼다.

**(b) `feedback-reporter` — per-팀 리네임 대신 단일 통합 reporter로 수렴 (특수 케이스)**
6-1. 스캔에서 발견된 **모든 `feedback-reporter.md`**(여러 팀 폴더에 흩어진)는 **리네임하지 않는다**(`name: feedback-reporter` 유지).
   대신 **전부 동일한 "통합 회고" 본문으로 덮어쓴다**(overwrite-identical). 어느 파일이 로드돼도 내용이 같으므로
   이름이 겹쳐도 무해해지고, 사실상 **1명의 통합 reporter**로 동작한다.
   - 이 스킬은 파일 삭제 도구가 없으므로 **동일 본문 덮어쓰기** 방식이다 → 재실행해도 결과가 같다(idempotent).
   - 통합 본문은 **스캔으로 파악한 "현재 조립된 팀 목록"을 회고 범위**로 박아, 전 팀 작업을 한 번에 회고하고
     **본부(team-architect)로 보내는 통합 `feedback.md` 1개**를 산출하도록 지시한다(아래 §2-2-c 본문 템플릿).

# 역할 — 조립 멀티팀 통합 회고·피드백 보고원

너는 이 프로젝트에 **조립된 전 팀(<조립된 팀 목록>)** 의 작업을 한 번에 회고해, 본부(team-architect)가
각 팀 정의·스킬·문서·템플릿을 바로 개선할 수 있도록 **통합 `feedback.md` 1개**를 산출한다.
디자인/코드/문서 등 직접 산출물은 만들지 않는다 — 회고만 하고 통합 피드백 파일 1개만 쓴다.

## 최우선 원칙
1. 이 회고는 가용 신호(git·세션·산출물 흔적) 기반 **best-effort**이며 세션 외 작업은 누락될 수 있다 — 머리말에 1줄 명시.
2. **추측 금지.** 근거 불확실은 `(확인 필요)`로 표기. 3. 칭찬·개선 모두 담는다(잘 된 점도 별도 섹션).

## 워크플로우
1. **범위 결정:** `feedback/`에서 가장 최근 통합 보고서를 찾아 *그 시점 이후*, 없으면 작업 전체.
2. **근거 수집(best-effort):** `git log --since="<최근 보고서 날짜>" --stat`(없으면 전체 이력) · 세션/대화 맥락 ·
   산출물 흔적. **팀별로** 무엇을 했는지 갈라 정리하되 불확실하면 `(확인 필요)`.
3. **작성:** 아래 표준 양식 — **"## 6. 팀별 회고"에 조립된 팀(<조립된 팀 목록>)마다 하위 섹션**을 두고,
   공통 이슈는 상단 우선순위 요약·이슈 상세에 모은다. 각 이슈에 *어떤 팀의 어떤 파일을 어떻게 바꾸나* 업데이트 대상을 적는다.
4. **저장:** `feedback/`에 날짜 포함 파일명으로 저장(예: `에이전트팀_통합피드백보고서_<주제>_YYYY-MM-DD.md`). 폴더 없으면 생성.
5. **보고:** 저장 경로 + 핵심 이슈 요약(표)을 보고한다.

## 3-2. 생성할 `/feedback-agents` 스킬 본문 (§2-4c — 여러 팀 조립 시 정규 통합 본문)

> 대상 프로젝트의 `.claude/skills/feedback-agents/SKILL.md`로 아래를 쓴다. 기존 팀 `feedback-agents/SKILL.md`
> 포맷(frontmatter `name: feedback-agents` + description + 트리거 `/feedback-agents` + 사용법 1줄 + `feedback-reporter`
> 위임 흐름)을 본떠 **"조립 멀티팀 통합 회고 → 본부 보고"** 로 작성한다. `<조립된 팀 목록>`은 스캔 결과로 채운다(예: `dev, qa, design`).

```markdown

name: feedback-agents
description: 이 프로젝트에 조립된 멀티팀(<조립된 팀 목록>)이 한 작업을 한 번에 회고해, 본부(team-architect)에 전달할 통합 피드백 보고서(feedback.md 1개)를 만듭니다. "회고", "피드백 보고서", "팀 개선점 정리" 요청 시 사용. 사용법 /feedback-agents [회고할 작업 주제(선택)]
---

# /feedback-agents — 조립 멀티팀 통합 회고·피드백 보고

통합 `feedback-reporter` 에이전트를 사용해 **이 프로젝트에 조립된 전 팀(<조립된 팀 목록>)이 한 작업을 한 번에
회고**하고, 본부(team-architect)가 각 팀 정의·스킬·문서를 바로 개선할 수 있도록 통합 `feedback.md` 1개를 산출합니다.

> 이 스킬은 `/team-assemble`가 생성한 **생성 아티팩트**입니다(팀을 추가/제거하고 다시 조립하면 회고 범위가 갱신됩니다).
> 스킬 없이도 **`feedback-reporter` 에이전트를 직접 호출**("feedback-reporter로 이번 작업을 회고해줘")하면 동일하게 동작합니다.

## 절차
1. **범위 결정:** `feedback/`에서 가장 최근 통합 보고서를 찾는다. 있으면 그 시점 이후, 없으면 작업 전체가 회고 범위.
2. 통합 `feedback-reporter`에게 위임: git 로그·세션 맥락·산출물 흔적을 best-effort로 수집 →
   **팀별로** 무엇을 했는지 갈라 `현상 → 영향 → 권장 업데이트(소속팀·대상 파일)` + 심각도(🔴/🟠/🟡)로 이슈 도출.
3. 표준 6섹션 양식(공통 이슈는 상단, **§6 팀별 회고에 조립된 팀마다 하위 섹션**)으로 통합 `feedback.md` 작성.
4. **저장:** `feedback/`에 날짜 포함 파일명으로 저장(폴더 없으면 생성). 본부(team-architect)에 전달한다.
5. 저장 경로와 핵심 이슈 요약(표)을 보고한다.

## 주의
- **best-effort 회고.** 가용 신호(git·세션·산출물 흔적) 기반이며 세션 외 작업은 누락될 수 있음을 보고서 머리말에 1줄 명시.
  근거 불확실 항목은 추측하지 말고 `(확인 필요)`로 표기.
- 회고·feedback 파일 1개만 작성한다 — 직접 산출물 작업·외부 변경(머지·배포 등)은 하지 않는다.

인자(`$ARGUMENTS`)가 있으면 회고할 작업 주제로 사용하고, 없으면 전체 범위로 회고한다.
```

## 5. 협업 핸드오프 패턴 (알려진 HQ 팀 — 조립된 팀만 사용)

현재 `.claude/agents/`에 **존재하는 팀에 해당하는 간선만** CLAUDE.md에 기입한다(없는 팀 흐름은 생략).

| 핸드오프 | 의미 |
|---|---|
| docs → dev | 기능/기술명세·요구사항 → 구현 |
| design → dev | 디자인 토큰/시스템 → 코드 정합(Code Connect) |
| dev → qa | 구현 완료 → 테스트 설계·실행·자동화 |
| qa → dev | 결함 트리아지 → 수정 |
| design → qa | 접근성(a11y)·UX → 검증 + 디자인 토큰/Figma·명세를 기준(SSOT)으로 **비주얼·디자인 정합**(토큰 준수·비주얼 리그레션·반응형) 검증 |
| docs → marketing | 기획·기능 → GTM/포지셔닝 핸드오프 |
| dev(레드팀) → dev(블루팀) | 침투 테스트 리포트 → 방어(퍼플팀 루프) |
| resume | 보통 독립(다른 팀과 핸드오프 적음) |

> 모르는(HQ 외) 팀이 섞여 있으면, 그 팀은 "독립"으로 두고 핸드오프는 사용자 보완을 권한다(추측 금지).
> **모델 A 팀(docs·design·marketing·resume·aiot·pm 등)이 핸드오프에 끼면** 그 팀은 에이전트 외에 스킬·`templates/`·
> 참조 문서·`.mcp.json`까지 있어야 자기완결로 동작한다 — §2-1b 자동 포트가 처리하며, HQ 레포가 없는 환경이면 수동 복사를 권한다(§1-1).

## 6. 안전·주의

- **마커 블록 밖은 절대 수정하지 않는다.** 사용자가 직접 쓴 CLAUDE.md 내용 보존.
- **에이전트 본문 자동 수정 금지.** `name:` 줄만 리네임하고, 본문 참조 변경은 사용자에게 알린다.
- **비밀값을 CLAUDE.md에 쓰지 않는다.** 파싱 중 발견한 토큰/키는 보고하지 말고 무시.
- **재실행 안전(idempotent).** 이미 리네임돼 충돌이 없으면 그대로 두고 블록만 다시 생성한다.
  `feedback-reporter`는 매번 동일한 통합 본문으로 덮어쓰므로(overwrite-identical) 재실행해도 결과가 같다.
  단일 팀만 있으면 덮어쓰지 않고 그 팀 전용 reporter를 보존한다.
- **통합 `/feedback-agents` 스킬과 (reporter가 없을 때의) `feedback-reporter` 에이전트는 생성 아티팩트**로,
  여러 팀 조립 시 매번 정규 통합 내용으로 재생성한다(idempotent). **단일 팀 조립 시엔 그 팀 전용 스킬을 보존**
  (덮어쓰지 않음)한다. 두 아티팩트 모두 **비밀값을 적지 않는다.**
- **MEMORY.md는 존재하면 덮어쓰지 않는다(누적 보존).** 새로 만들 때만 §3-1 시드를 넣는다. 재실행 시
  이미 있으면 그대로 둔다(사용자·이전 세션이 쌓은 크로스팀 학습을 잃지 않게). 비밀값은 시드에도 적지 않는다.
- **자동 포트(§2-1b)는 덮어쓰지 않는다.** 대상에 동일명 스킬 폴더·템플릿·참조 문서·서버 키가 이미 있으면
  보존하고 보고만 한다(diff 확인 권유). 복사 중 실값 비밀이 보이면 그 항목은 중단·보고(비밀값 레포 금지).
- **재배포(팀 업데이트)는 §1-2 체크리스트를 따른다** — 오배치·협업 CLAUDE.md 덮어쓰기는 멀티팀 운영 규칙 전체를 무력화한다. 스캔 중 *팀 폴더와 소속이 어긋난 에이전트*(예: dev/ 폴더의 docs 에이전트 다수)가 보이면 오배치 의심으로 보고한다.
- **조립·재배포 중 시크릿 노출이 보이면 즉시 보고한다.** 대상 프로젝트에서 `.env`·`serviceAccountKey.json`·
  `credentials*.json` 류 평문 시크릿 파일이 눈에 띄면(스캔 중 우연 발견 포함) 조립 작업보다 먼저 사용자에게
  보고하고 revoke·재발급을 권고한다(실사례 2026-06-11: 투입 중 `.env`의 API 키 평문 발견 → Sev1).
- 파싱 불가/`name` 누락 파일은 **건너뛰고 경고**에 남긴다(임의로 이름을 만들지 않는다).
- 같은 팀 폴더 안의 정상 중복(같은 파일 2개)은 사용자에게 확인을 권한다.
